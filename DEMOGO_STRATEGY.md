# DemoGo 战略定位与阶段目标

更新时间：2026-05-11

## 1. 核心定位

DemoGo 不是传统云服务器、低代码平台或完整 DevOps 平台。

DemoGo 的目标定位是：

> AI 生成产品的发布、体检、试用和数据回收平台。

一句话表达：

> 让 AI 生成的小产品，真正被别人打开、试用，并把反馈和数据收回来。

当前产品已经完成可用 MVP 雏形，下一阶段要从“能发布静态 Demo”升级为“准商业化试用 MVP”。

## 2. 为什么需要升级定位

当前 v0.1.6 已经具备注册、上传、检测、发布、生成链接和 Demo 管理能力，但仍偏向静态发布工具。

真实用户的高频需求不止是宣传页，还包括：

- 活动报名页；
- 预约体验页；
- 产品内测申请页；
- 公司官网联系表单；
- 简单订单意向页；
- 简单问卷反馈；
- 带基础后台查看能力的轻量 MVP。

这些场景的共同点是：页面要能打开，数据也要能回来。

因此 DemoGo 的产品价值需要从“给一个链接”升级为：

- 判断项目能不能发布；
- 判断发布后哪些功能会失效；
- 识别表单、接口和数据风险；
- 生成给 AI 编程工具的修复指令；
- 提供标准表单数据托管能力；
- 在用户端查看和导出提交数据。

## 3. 目标用户

| 用户类型 | 典型场景 | 核心需求 |
|---|---|---|
| AI 编程工具用户 | 用 Cursor、Trae、Codex、Claude Code 等生成小产品 | 快速发布和分享 |
| 产品经理/创业者 | 做 MVP、报名页、预约页、内测页 | 验证需求并收集数据 |
| 外包/交付人员 | 给客户展示阶段成果 | 低成本生成演示链接 |
| 培训/课程用户 | 展示作业或作品 | 免部署门槛 |
| 中小企业 | 简单官网、活动页、留资页 | 表单数据可查看可导出 |

## 4. 不做什么

短期不要把 DemoGo 做成万能云开发平台。

当前阶段不优先支持：

- 任意后端服务长期运行；
- 用户自带数据库托管；
- Docker 应用发布；
- 复杂用户权限系统；
- 真实支付和库存订单系统；
- 多角色企业级后台；
- 高并发生产环境应用托管。

这些能力可以作为中长期方向，但不应进入 v0.1.x 和 v0.2.x 的核心范围。

## 5. 产品边界

建议采用三层边界：

| 层级 | 能力 | 阶段 |
|---|---|---|
| 静态/前端发布 | HTML、React、Vue、Vite 等前端项目发布为链接 | 已有，继续优化 |
| AI 项目体检 | 识别项目类型、表单、API、构建风险，生成修复建议 | v0.1.7 起建设 |
| 标准数据托管 | 活动报名、预约、留言、反馈、订单意向等表单数据保存和导出 | v0.2 核心 |

核心原则：

> 不自动解析和接管所有客户自定义 API，而是提供 DemoGo 标准数据接口，让客户项目按规则接入。

## 6. 阶段目标

### 阶段 A：可用 MVP，当前状态

目标：证明 DemoGo 可以发布 AI 生成的前端 Demo。

已实现：

- 用户注册登录；
- 上传 zip；
- 项目检测；
- 静态和常见前端源码项目发布；
- Demo 链接生成；
- Demo 下线、恢复、删除、更新；
- 基础访问统计；
- 管理后台。

### 阶段 B：准商业化试用 MVP

目标：支持小范围真实用户试用，能看懂问题、能人工运营、数据结构稳定。

应完成：

- 数据库和正式表结构；
- 发布体验优化；
- AI 项目体检基础版；
- 支持边界清晰提示；
- 管理后台套餐调整；
- 备份和上线检查机制；
- 域名、HTTPS 等对外可信能力。

### 阶段 C：数据回收 MVP

目标：不仅能发布页面，还能收集报名、预约、留言、订单意向等数据。

应完成：

- 表单托管；
- 表单字段定义；
- 提交数据保存；
- 用户端查看、搜索、导出；
- AI 生成接入指令；
- 后续飞书/腾讯文档集成预留。

### 阶段 D：商业化验证

目标：验证用户是否愿意为 Demo 发布、体检、数据回收付费。

应完成：

- Free/Lite/Pro 套餐正式化；
- 人工开通和套餐变更记录；
- 订单系统雏形；
- 真实访问统计；
- 用量限制；
- 客户反馈闭环。

## 7. 核心差异化

DemoGo 不应该和 Vercel、Netlify 正面比部署能力。

DemoGo 的差异化是：

| 普通部署工具 | DemoGo |
|---|---|
| 给代码一个链接 | 看懂 AI 生成项目，给链接和体检报告 |
| 面向开发者 | 面向 AI 编程用户和非专业开发者 |
| 默认用户懂构建、API、后端 | 主动解释哪里会失败、怎么修 |
| 只托管页面 | 逐步支持数据回收和轻量后台 |
| 失败日志偏技术 | 生成可理解修复建议和 AI 编程工具指令 |

## 8. 近期判断

下一阶段不应只做页面文案优化，也不应马上支持完整后端应用。

更合理的方向是：

> 数据库正规化 + AI 项目体检基础版 + 发布体验优化 + 为表单托管预留架构。

这能让 DemoGo 从“极简 MVP”走向“可试用、可运营、可商业验证”的产品。
