# DemoGo v0.1.12 版本计划：试运营体验重设计版

更新时间：2026-05-12

## 1. 版本定位

v0.1.12 的目标不是继续零散补按钮，而是把 DemoGo 从“功能能跑”推进到“可以拿给真实用户试运营”。

一句话定位：

> 重新设计首页、用户端和管理后台，让用户看得懂、用得顺，运营人员能处理真实试用线索。

本版本同时启动前端现代化迁移：

```text
React + Vite + TypeScript
```

因此 v0.1.12 的完整定位调整为：

```text
前端现代化重构 + 三端体验重设计版
```

v0.1.11 已经完成发布、套餐申请、发布记录、反馈和管理后台处理的基本闭环，但页面体验仍然明显偏 MVP：

- 首页还像早期介绍页，缺少面向试运营的产品信任感；
- 用户端信息组织不清晰，申请、记录、额度之间关系不够直观；
- 管理后台运营待办不够突出；
- Free / Lite / Pro 命名需要完全统一；
- 升级申请状态、拒绝原因、处理记录没有形成清楚列表；
- 页面视觉密度、层级、操作路径和国际成熟平台仍有明显差距。

本版本不再继续在 `index.html`、`app.html`、`admin.html` 单文件里堆功能，而是新建 `web/` 前端工程。构建产物仍部署为静态文件，线上路径保持兼容。

## 2. 参考对象

本版本参考国际成熟平台的“信息组织方式”，不照抄视觉。

### Vercel

参考点：

- Project 作为核心对象；
- 每个 Project 下有 Deployments、Domains、Usage、Logs 等结构；
- Dashboard 支持项目视图、部署视图、用量视图和搜索；
- 项目详情强调当前状态、最近部署、访问入口和下一步动作。

DemoGo 借鉴：

- Demo 应该成为用户端核心对象；
- 每个 Demo 应有状态、链接、版本、发布记录、转发文案、检测报告；
- 列表和详情要分清，不要把所有动作堆在一个卡片里。

### Netlify

参考点：

- Site 和 Deploy 是核心对象；
- Forms 有独立入口和提交数据列表；
- 表单提交支持查看、筛选、导出和状态处理；
- 表单能力和部署能力在同一站点下组织。

DemoGo 借鉴：

- 后续表单托管应挂在 Demo 下；
- 当前 v0.1.12 先把“升级申请、反馈、发布记录”做成清晰列表；
- 记录类功能必须列表化，不适合只用一句提示。

### Railway / Render

参考点：

- Deployment 是有状态的流程；
- 部署有 Building、Deploying、Failed、Active 等状态；
- Render 强调 Events timeline，部署、跳过、取消、回滚都进入事件流；
- 操作记录和状态变化是运维后台的重要组成部分。

DemoGo 借鉴：

- 发布、更新、下线、恢复、删除、升级申请、反馈处理都应该逐步事件化；
- 用户端展示“本月发布记录”；
- 管理后台展示“运营待办”和“最近事件”。

## 3. 本版核心原则

### A. 套餐命名统一

套餐名称正式统一为：

```text
Free
Lite
Pro
```

规则：

- 前端、用户端、管理后台都显示 Free / Lite / Pro；
- 不再把“免费体验”“轻量版”作为套餐名称；
- 中文只作为说明文案，例如“适合初次试用”“适合客户演示”；
- 后端技术字段继续使用 `free` / `lite` / `pro`，不展示给用户。

### B. 卡片和列表各司其职

卡片用于“选择”和“概览”：

- 套餐选择；
- 当前套餐概览；
- 首页能力介绍；
- 管理后台关键指标。

列表用于“记录”和“流程”：

- Demo 列表；
- 发布记录；
- 升级申请；
- 反馈记录；
- 后台运营待办；
- 后续表单提交数据。

这是 v0.1.12 的界面设计原则。

## 4. 用户端改造

### A. 信息架构

建议用户端改为：

1. 概览；
2. 我的 Demo；
3. 上传发布；
4. 套餐与额度；
5. 我的升级申请；
6. 本月发布记录；
7. 反馈问题。

### B. 概览

进入用户端第一屏应展示：

- 当前套餐：Free / Lite / Pro；
- 在线 Demo 使用量；
- 本月发布/更新使用量；
- 最近一个 Demo；
- 待处理升级申请；
- 主要下一步操作：上传 Demo / 查看 Demo / 申请升级。

### C. 我的 Demo

Demo 列表要更像产品控制台，而不是简单链接集合。

每条 Demo 显示：

- Demo 名称；
- 状态；
- 访问链接；
- 版本；
- 访问次数；
- 到期时间；
- 最近更新时间；
- 操作：复制链接、复制转发文案、更新、下线、恢复、删除。

后续可以进入 Demo 详情页，本版先不强制拆独立页面。

### D. 套餐与额度

套餐卡片统一显示：

| 套餐 | 在线 Demo | 发布/更新 | Demo 保留 | 场景说明 |
|---|---:|---:|---:|---|
| Free | 1 | 3 次/月 | 7 天 | 初次试用 |
| Lite | 3 | 20 次/月 | 30 天 | 客户演示 |
| Pro | 10 | 60 次/月 | 30 天 | 持续试用 |

交互规则：

- 当前套餐高亮；
- Free 可申请 Lite / Pro；
- Lite 可申请 Pro；
- Pro 不能申请降级；
- 有待处理申请时，禁用重复提交。

### E. 我的升级申请

新增列表，不再只显示一句最近状态。

字段：

| 字段 | 说明 |
|---|---|
| 申请套餐 | Lite / Pro |
| 状态 | 待处理 / 已开通 / 已拒绝 |
| 提交时间 | 用户提交时间 |
| 处理时间 | 管理员处理时间 |
| 管理员说明 | 拒绝原因或处理说明 |

用户看到“已有待处理申请”时，必须能在这里看到是哪一条。

### F. 本月发布记录

继续保留并优化为列表：

| 时间 | Demo | 操作 | 版本 | 额度影响 |
|---|---|---|---|---|
| 05-12 15:20 | 活动报名页 | 发布 | V1 | 计入 |
| 05-12 15:40 | 活动报名页 | 更新 | V2 | 计入 |

### G. 复制文案

保留 v0.1.11 的原则：

- 点击即复制；
- 不跳转；
- 不默认弹出文本框；
- 成功后按钮显示“已复制”；
- 只有失败才显示备用文本框。

## 5. 首页落地页改造

首页不是功能说明页，而是转化页。它要让访问者马上意识到：

> 我用 AI 做出来的页面或产品雏形，终于可以很快发给别人真实试用了。

### A. 首页要打中的核心痛点

目标用户不是专业部署工程师，而是产品经理、运营、创业者、售前、业务人员和 AI 编程工具用户。

他们的问题通常不是“我需要一个云平台”，而是：

- AI 已经生成了页面，但只能停在本地；
- 发截图和录屏没有说服力，别人不能真正点击体验；
- 发 zip 包给别人，对方不会打开，也不愿意折腾；
- 用 Vercel / Netlify 这类工具，对非技术用户仍然偏复杂；
- 国外部署平台虽然强大，但面向国内客户、领导、同事、投资人或普通试用用户时，访问链路可能不稳定；
- 自己买服务器、配置域名解析、SSL、Nginx、备案和运维，对非技术用户来说成本太高；
- AI 生成的项目经常藏着问题，比如本地 `/api/register`、表单不能保存、缺少构建命令；
- 好不容易生成链接，还不知道怎么组织一段转发给客户或试用用户的话。

DemoGo 首页要回答：

- 为什么需要 DemoGo；
- DemoGo 解决什么痛点；
- 和普通部署工具有什么不同；
- 用户现在可以怎么开始。

### B. 首屏文案方向

首屏不再先讲“发布平台”，而是先讲痛点。

建议主标题：

```text
AI 做出了产品雏形，
别再卡在部署和打开这一步。
```

建议副标题：

```text
DemoGo 帮你绕开买服务器、配域名、研究部署平台这些麻烦，
把 AI 生成的网页、活动页和产品原型，
快速变成可打开、可转发、可试用的 Demo 链接。
```

补充说明：

```text
上传 zip 项目包，DemoGo 会自动做基础体检，
提示入口页面、表单字段和本地 API 风险，
生成更适合国内试用场景的访问链接和可复制的转发文案。
```

首屏按钮：

```text
立即生成试用链接
查看适用场景
```

首屏右侧或背景建议展示“真实产品界面感”的控制台预览，而不是抽象插画。

### C. 场景对比模块

用强对比让用户意识到 DemoGo 的价值。

没有 DemoGo：

- 发截图，别人只能看，不能点；
- 发录屏，反馈不具体；
- 发 zip，对方打不开；
- 发本地地址，别人访问不了；
- 国外平台链接在国内试用时可能不稳定；
- 自己部署要处理服务器、域名、SSL、备案和运维；
- 项目里有表单或 API 问题，发布后才发现不能用。

使用 DemoGo：

- 上传 zip；
- 自动检测能不能发布；
- 生成更适合国内试用场景的访问链接；
- 一键复制转发文案；
- 发给别人真实试用；
- 提前知道表单/API 风险。

### C1. 和国外部署平台、自建服务器的差异化

这一块要成为首页“为什么选择 DemoGo”的核心内容之一。

建议表达：

```text
Vercel、Netlify 这类平台很强，适合开发者做正式工程部署。
但很多国内试用场景的真正问题不是“能不能部署”，而是客户、同事、领导、投资人能不能稳定打开。
DemoGo 面向的是 AI 生成项目的快速试用验证：上传项目包，生成国内用户更容易打开的试用链接。
```

对比表：

| 方式 | 更适合谁 | 主要麻烦 |
|---|---|---|
| Vercel / Netlify 等国外平台 | 开发者、正式工程部署 | 需要理解 Git、构建、域名、环境变量；国内试用访问可能不稳定 |
| 自己买服务器部署 | 技术团队 | 服务器、Nginx、SSL、域名解析、备案、运维 |
| 发截图 / 录屏 / zip | 临时沟通 | 对方不能真实点击试用，反馈价值低 |
| DemoGo | AI 产品验证、客户演示、活动页试运营 | 上传 zip，自动检测，生成试用链接和转发文案 |

一句话优势：

```text
DemoGo 把部署、服务器、域名、检测和转发这些麻烦，压缩成一次上传。
```

合规边界：

```text
DemoGo 适合测试、演示和试用验证。如果后续绑定独立域名、做正式商用网站或长期运营，仍需根据实际业务办理域名、备案和相关合规事项。
```

### D. 三步流程模块

标题：

```text
从 AI 生成项目到可试用链接，只需要三步
```

步骤：

1. 上传项目包；
2. DemoGo 自动体检；
3. 生成链接并转发试用。

每一步要用非技术语言：

- 不讲“构建产物”“路由代理”；
- 讲“上传”“检查”“生成链接”。

### E. 差异化模块：不只是发布链接

标题：

```text
不只是生成链接，还会在发布前做基础体检，提示可能影响试用的风险。
```

能力表达：

- 检测入口页面；
- 检测表单字段；
- 检测本地 API；
- 提示页面能打开但表单提交、数据保存等功能可能不可用的风险；
- 生成给 AI 编程工具的修改提示；
- 生成适合转发给试用用户的文案。

这部分是 DemoGo 区别于普通静态部署工具的关键。当前是规则体检基础版，不要在首页表达成大模型 AI 判断或完整业务功能测试。

### F. 适用人群模块

标题：

```text
适合这些正在用 AI 做产品的人
```

人群：

- 用 Cursor / Claude Code / Codex / 通义灵码生成页面的人；
- 想快速验证产品原型的创业者；
- 需要给客户演示方案的售前和销售；
- 做活动报名页、预约页、咨询页的运营人员；
- 想让同事、客户、投资人快速试用页面的产品经理。

### G. 典型场景模块

标题：

```text
这些 Demo，最适合先用 DemoGo 发出去试试
```

场景：

- AI 生成的产品原型；
- 活动报名页；
- 预约咨询页；
- 产品介绍页；
- 客户演示页；
- 内部试用页面；
- 课程、服务、方案展示页。

### H. 支持边界模块

这一块要清楚，但不要放在最前面吓退用户。

表达方式：

```text
当前 DemoGo 最适合发布前端页面和静态 Demo。
如果项目依赖自定义后端、数据库或支付，DemoGo 会在检测时提示风险。
```

支持：

- 静态 HTML；
- dist / build 前端项目；
- 常见 React / Vue / Vite 源码项目；
- 活动页、展示页、报名页前端。

暂不支持：

- 完整后端服务；
- 数据库；
- Docker；
- 真实支付；
- 自动接管任意自定义 API。

### I. 套餐模块

套餐名称统一为：

- Free；
- Lite；
- Pro。

套餐文案要轻，不要像复杂 SaaS 计费页。

重点告诉用户：

- Free 适合先试；
- Lite 适合客户演示；
- Pro 适合持续验证多个 Demo。

### J. 结尾 CTA

结尾要回到痛点。

建议文案：

```text
别让 AI 做出来的产品停在本地。
现在把它变成一个真正能打开、能转发、能试用的 Demo 链接。
```

按钮：

```text
开始发布 Demo
```

### K. 首页语气要求

首页必须像直接给用户看的页面：

- 少讲技术名词；
- 少讲平台架构；
- 多讲用户场景；
- 多讲试用、转发、反馈、验证；
- 支持边界要清楚，但不要一开始就堆限制；
- 让用户感觉“这就是帮我解决 AI 作品发不出去的问题”。

## 6. 管理后台改造

### A. 信息架构

建议管理后台改为：

1. 运营概览；
2. 待办事项；
3. Demo 管理；
4. 用户列表；
5. 升级申请；
6. 反馈列表；
7. 最近事件；
8. 风控与系统状态。

### B. 运营概览

第一屏展示：

- 注册用户；
- 在线 Demo；
- 今日发布；
- 待处理升级申请；
- 待处理反馈；
- 近 7 日访问趋势，若数据不足先显示占位。

### C. 待办事项

把最需要运营人员处理的内容放上来：

- 待处理升级申请；
- 待处理反馈；
- 发布失败或风险提示；
- 即将到期 Demo，后置可做。

### D. 升级申请列表

必须展示处理说明。

字段：

| 用户 | 当前套餐 | 申请套餐 | 联系方式 | 说明 | 状态 | 提交时间 | 处理时间 | 处理说明 | 操作 |
|---|---|---|---|---|---|---|---|---|---|

操作：

- 开通；
- 拒绝；
- 拒绝原因页面内输入；
- 处理后列表直接展示原因。

### E. 用户列表

只读查看：

- 用户邮箱；
- 当前套餐；
- Demo 数；
- 在线 Demo；
- 最近发布；
- 注册时间；
- 最近升级申请状态。

不允许直接改套餐，套餐开通只走升级申请。

## 7. 后端与数据

本版优先复用现有数据表：

- `users`
- `demos`
- `demo_versions`
- `feedback`
- `plan_upgrade_requests`
- `audit_logs`

建议后端改造：

- `publicPlanRequest` 必须返回 `adminNote`、`handledAt`、`handledBy`；
- 用户端升级申请接口返回最近申请列表；
- 管理后台升级申请列表展示处理说明；
- 统一套餐展示名函数，返回 Free / Lite / Pro；
- 增加 smoke test：拒绝申请后，用户端接口能看到拒绝原因。

本版不新增订单表，不做支付。

## 8. 架构演进

v0.1.12 的架构演进重点：

- 新增 `plan-display-service` 或在现有 service 中统一套餐展示名；
- 将升级申请相关展示逻辑收敛到 service；
- 前端提取套餐配置对象，避免 Free / Lite / Pro 文案散落；
- 管理后台和用户端都使用同一套状态标签文案；
- 为后续页面组件化做准备，但本版仍保持静态 HTML。

## 9. 不纳入本版本

- 在线支付；
- 正式订单系统；
- 表单托管正式版；
- 大模型 AI 体检；
- Docker / 后端项目托管；
- 独立 Demo 详情页复杂化；
- 完整前端框架迁移；
- 微服务拆分。

## 10. 验收标准

用户端：

- Free / Lite / Pro 命名统一；
- 用户能看到升级申请列表；
- 待处理申请清楚展示，不能重复提交；
- 拒绝申请后能看到管理员说明；
- 本月发布记录是列表；
- 复制文案仍然一键复制；
- 页面第一屏能看懂当前状态和下一步。

首页：
