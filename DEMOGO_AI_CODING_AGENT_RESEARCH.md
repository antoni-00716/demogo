# DemoGo AI 编程 Agent 产物调研与部署能力路线

更新时间：2026-05-15

## 1. 结论先行

DemoGo 后续能力不能只按“技术实现容易程度”排序，而要按 AI 编程工具真实生成的项目类型来排序。

当前主流 AI 编程工具已经不只是生成静态页面，而是在生成“可运行产品原型”。这些原型常见形态包括：落地页、前端单页应用、报名/预约/表单页、Dashboard、CRM/SaaS 原型、电商/订单页、AI 工具、小游戏、全栈应用、Node/Python 后端项目、Next.js/SSR 项目等。

因此，DemoGo 的产品定位应从“静态页面发布工具”升级为：

> AI 生成产品的试用发布平台。

短期不能直接做成完整 Railway / Render / Zeabur，但也不能停留在只能发布静态 HTML。最合理的路线是：

1. 先把静态页面和前端构建项目做到稳定、可解释、可恢复；
2. 立即补“表单/报名/预约数据托管”，解决最高频的真实业务闭环；
3. 把同步部署改成异步部署任务，解决构建超时和用户体验问题；
4. 后续再引入 Docker 构建隔离、Node 后端运行、数据库分配、SSR 支持。

## 2. 调研对象

本次调研对象分三类。

### 2.1 AI 编程 Agent

这些工具最接近 DemoGo 的真实项目来源，因为它们会生成真实代码仓库。

| 工具 | 当前特征 | 对 DemoGo 的意义 |
|---|---|---|
| Codex | 可读代码、改代码、运行代码，可在云环境后台工作，并可从 GitHub 创建 PR | 产物更接近真实工程仓库，可能包含构建脚本、测试、环境变量、后端代码 |
| Claude Code | 终端/IDE/Desktop/Web 中的 agentic coding 工具，可读代码、改文件、运行命令、创建 PR、接 MCP | 产物可能包含多文件、多模块、后端、数据库、CI 配置 |
| Cursor | IDE Agent 工作流，适合用户在本地迭代项目 | 产物多为本地项目目录，用户可能直接压缩上传 |
| OpenCode | 开源终端编码 Agent，面向本地代码编辑、测试和多模型接入 | 产物同样是本地工程仓库 |
| GitHub Copilot coding agent | 在 GitHub 的临时开发环境里改代码、跑测试、走 PR 流程 | 产物偏标准 GitHub 仓库，未来适合做 GitHub 导入 |
| Windsurf / Gemini CLI / Gemini Code Assist | IDE/CLI/云开发环境里的 Agent 工作流 | 产物形态与 Cursor、Claude Code、Codex 类似 |
| Hammers Agent | 用户自有/关注工具，公开资料不足 | 需要按实际生成项目样本补充判断 |
| OpenClaw | 更偏通用自主 Agent/工作流 Agent，不是纯编码 Agent | 未来可能驱动代码生成和部署，但当前不应作为核心项目类型依据 |

### 2.2 AI 应用生成平台

这些平台不是 DemoGo 的直接竞品，但非常适合观察“用户最想生成什么”。

| 平台 | 观察价值 |
|---|---|
| Bolt | 用户会生成全栈应用、商业页面、Dashboard、CRM、电商、表单等 |
| Lovable | 强调 Supabase、Stripe、GitHub 等集成，说明数据库、认证、支付需求普遍 |
| v0 | 模板集中在 Landing Page、Dashboard、Login、E-commerce、AI、Agents、Components |
| Replit Agent | 明确覆盖生产力工具、电商、游戏、AI Chatbot，并包含数据库、文件存储、认证、密钥管理 |
| Firebase Studio | 支持全栈 AI 应用、API、后端、前端、移动端、Next.js、React、Vue、Node、Python Flask 等 |

### 2.3 DemoGo 当前真实样本

目前已经遇到的真实样本包括：

- 纯 HTML 页面；
- 已构建的 `dist/index.html` 或 `build/index.html`；
- React/Webpack/Vite 源码项目；
- 有表单和 `/api/register`、`/api/submit` 等本地 API 调用的项目；
- 用户上传 `.zip`、`.tar.gz`、`.tgz`；
- 用户项目发布成功但页面空白，根因是源码项目被误按静态壳页面发布；
- 构建时间较长时 `/api/deploy` 同步请求被 Nginx 超时中断。

这些样本说明：DemoGo 已经进入“发布真实 AI 编程产物”的阶段，不能再只把它当简单静态文件上传工具。

## 3. AI 编程工具常见产物类型

### 3.1 静态落地页 / 官网 / 作品集

典型项目：

- 活动宣传页；
- 产品介绍页；
- 公司官网；
- 个人作品集；
- 等候名单页面；
- 产品试用申请页。

技术特征：

- 可能是单个 `index.html`；
- 也可能是 React/Vue/Vite 构建后的 `dist`；
- 通常不需要后端；
- 可能包含联系表单或试用申请表单。

DemoGo 当前支持度：高。

下一步重点：

- 发布成功率和错误提示继续优化；
- 首页和发布页明确告诉用户适合上传什么；
- 如果检测到表单，要提示“页面可打开，但数据保存需要 DemoGo 表单托管”。

### 3.2 前端 SPA 工具 / 小游戏 / Dashboard

典型项目：

- 计算器；
- AI 小工具前端；
- 简单小游戏；
- 数据看板；
- 管理后台界面原型；
- 任务管理面板。

技术特征：

- 常见 React/Vue/Vite/Webpack；
- 需要 `npm install` 和 `npm run build`；
- 发布产物通常是 `dist` 或 `build`；
- 可能依赖浏览器本地存储；
- 如果调用本地 `/api/*`，静态发布后相关功能会失效。

DemoGo 当前支持度：中。

当前问题：

- 源码构建是同步执行，容易超时；
- 构建在宿主机执行，长期有安全隐患；
- 对复杂依赖、私有包、环境变量支持不足；
- 不支持运行后端进程。

下一步重点：

- v0.1.17 需要把“可构建前端项目”标为实验能力；
- 构建失败要输出可理解的原因和给 AI 编程工具的修复指令；
- v0.2.0 前后改为异步部署任务。

### 3.3 表单 / 报名 / 预约 / 线索收集

典型项目：

- 培训报名页；
- 活动预约页；
- 试用申请页；
- 联系我们；
- 问卷反馈；
- 简单订单意向页。

技术特征：

- 前端看起来简单；
- 真实价值在数据提交和后台查看；
- AI 工具经常会生成 `fetch("/api/submit")`、`fetch("/api/register")`；
- 用户通常以为“上传后表单就能用”，但当前 DemoGo 还不会自动保存这些数据。

DemoGo 当前支持度：低，但应优先做。

建议：

这是 DemoGo 下一阶段最高优先级能力。原因是它比完整后端简单很多，但商业价值明显更高。

MVP 做法：

1. 上传时检测表单字段和本地 API 调用；
2. 允许用户为某个 Demo 创建“数据表”；
3. DemoGo 生成标准提交接口；
4. 用户把接口地址交给 AI 编程工具，让它把表单提交改到 DemoGo；
5. DemoGo 用户端展示提交记录；
6. 支持 CSV 导出；
7. 管理后台可看到表单数量、提交量、异常提交。

这不是完整后端托管，但可以覆盖大量真实业务验证场景。

### 3.4 SaaS / CRM / 内部工具原型

典型项目：

- 客户管理；
- 订单管理；
- 课程管理；
- 项目管理；
- 员工/会员管理；
- 简单审批后台。

技术特征：

- 一般需要登录；
- 需要数据库；
- 需要 CRUD API；
- 可能有管理权限；
- 可能有多个页面和复杂状态。

DemoGo 当前支持度：低。

判断：

这类需求很常见，但不适合马上完整支持。原因是它已经从“页面发布”进入“应用托管”，涉及后端运行、数据库、安全隔离、权限、日志和备份。

MVP 替代方案：

- 先支持前端原型发布；
- 明确提示“当前仅支持界面演示，不托管业务后台”；
- 对简单表单数据先走 DemoGo 表单托管；
- 后续进入 Docker 后端运行阶段再逐步支持。

### 3.5 电商 / 订单 / 支付

典型项目：

- 商品展示页；
- 购物车；
- 简单订单；
- 支付按钮；
- 订阅购买页。

技术特征：

- 静态商品展示可以发布；
- 订单和支付必须有后端；
- 涉及支付回调、安全校验、订单状态、隐私数据；
- Stripe/支付宝/微信支付等接入都不能靠静态页面完成。

DemoGo 当前支持度：低。

建议：

短期只支持“商品展示/订单意向收集”，不支持真实支付闭环。

可以先做：

- 意向订单表单；
- 联系方式收集；
- 订单线索导出；
- 提示用户“真实支付暂不支持”。

真实支付应放到更后面，并作为商业化能力慎重设计。

### 3.6 AI 工具 / Chatbot / Agent 页面

典型项目：

- AI 文案工具；
- AI 聊天页面；
- 图片生成页面；
- 简历生成器；
- Agent 控制台。

技术特征：

- 前端页面可以发布；
- API Key 不能放在浏览器端；
- 需要服务端代理；
- 可能需要流式输出；
- 可能需要用户额度控制。

DemoGo 当前支持度：低。

建议：

短期不能支持“直接上传后可用的 AI 后端工具”。否则容易诱导用户把 API Key 写进前端，风险很高。

可先做：

- 检测 `.env`、API Key 字样、OpenAI/Claude/Gemini 等调用；
- 明确提示“AI 接口需要后端代理，当前 DemoGo 不托管自定义后端”；
- 后续支持“DemoGo 托管 AI Proxy”时再开放。

### 3.7 SSR / Next.js / Nuxt / Remix

典型项目：

- Next.js App Router；
- 带 API routes 的 Next.js 项目；
- Nuxt 网站；
- SSR SaaS 模板。

技术特征：

- 不是简单 `dist/index.html`；
- 需要 Node 运行时；
- 可能需要 `.next`、server bundle、环境变量；
- 需要进程管理和反向代理。

DemoGo 当前支持度：低。

建议：

短期不要伪装支持。检测到 Next/Nuxt/Remix 时，应分清：

- 如果项目能静态导出，则提示用户导出静态产物再上传；
- 如果需要 SSR，则明确提示“当前版本暂不支持运行时项目”。

### 3.8 Node / Python 后端项目

典型项目：

- Express API；
- FastAPI；
- Flask；
- Django；
- NestJS；
- 小型全栈应用。

技术特征：

- 需要启动进程；
- 需要端口；
- 需要日志；
- 需要健康检查；
- 需要资源限制；
- 需要容器隔离；
- 可能需要数据库。

DemoGo 当前支持度：低。

建议：

这是长期方向，但不应在 v0.1.x 草率实现。正确路线是：

1. Docker 构建隔离；
2. 后端容器运行；
3. Nginx/Caddy 路由；
4. 环境变量管理；
5. 日志查看；
6. 资源限制；
7. 自动停止/清理；
8. 数据库连接或数据库分配。

## 4. DemoGo 当前能力边界

截至 v0.1.16，DemoGo 当前真实能力如下。

### 4.1 已支持

- 用户注册登录；
- Free / Lite / Pro 套餐；
- 升级申请；
- Demo 创建、更新、下线、恢复、删除；
- `.zip` 上传；
- `.tar.gz` 和 `.tgz` 上传；
- 纯 HTML 静态发布；
- `dist/index.html` 发布；
- `build/index.html` 发布；
- 部分 React/Vite/Webpack 源码项目自动构建；
- 识别 `package.json` 和 `scripts.build`；
- 识别表单字段；
- 识别本地 API 调用；
- 拦截 `.env` 等敏感文件；
- 忽略 `node_modules`、部分 AI 工具内部目录；
- 发布记录基础数据；
- MySQL 业务数据存储；
- Nginx 反向代理；
- 基础管理后台。

### 4.2 部分支持但需要标注为实验能力

- 源码项目自动构建；
- 构建产物目录识别；
- 表单/API 风险提示；
- 发布事件日志；
- 项目检测报告。

这些能力能用，但还不够平台级稳定。

### 4.3 当前不支持

- 自定义后端 API 托管；
- Node/Python 后端运行；
- SSR 运行时；
- Next.js / Nuxt 服务端渲染；
- 数据库自动分配；
- 用户自定义环境变量；
- 支付回调；
- 后台任务；
- WebSocket/实时通信；
- 服务端 AI Proxy；
- 自定义域名；
- HTTPS 正式域名体系；
- GitHub 导入；
- Docker 隔离构建的生产级流程；
- 异步部署任务。

## 5. 当前关键问题

### 5.1 同步部署导致超时

现在 `/api/deploy` 会在一个请求里完成上传、解压、检测、安装依赖、构建、发布。只要源码项目构建时间稍长，Nginx 就可能超时，用户端就可能收到 HTML 错误页，然后前端按 JSON 解析，出现：

```text
Unexpected token '<', "<html> <h"... is not valid JSON
```

这不是简单 Nginx 配置问题，而是部署架构问题。

正确方向：

```text
上传项目
  -> 创建部署任务
  -> 立即返回 deploymentId
  -> 后台继续执行检测/构建/发布
  -> 前端轮询任务状态和日志
  -> 成功后展示访问链接
```

### 5.2 当前构建隔离不足

源码项目构建如果直接在宿主机执行，长期风险较高：

- 依赖安装可能执行脚本；
- 可能消耗大量 CPU/内存；
- 可能访问系统文件或网络；
- 构建失败原因复杂；
- 多用户并发时会相互影响。

短期可以作为受限实验能力，但长期必须进入 Docker 隔离。

### 5.3 用户对“支持什么”没有清晰预期

用户会以为：

- 有表单就能保存数据；
- 有 `/api/submit` 就能提交；
- 有后台代码就能运行；
- 有 Next.js 就能自动上线；
- 有订单页就能收款。

如果 DemoGo 不明确提示边界，用户会认为是平台故障。

所以 v0.1.17 起应强化“项目检测报告”和“支持能力说明”。

### 5.4 打包入口仍有摩擦

AI 编程 Agent 产出的通常是一个项目目录或 Git 仓库，DemoGo 当前入口是压缩包上传。

这对非技术用户有两个问题：

- 不知道应该压缩哪个目录；
- 不知道应该上传源码还是 `dist/build`。

短期要在用户端给出更明确的打包指引。

中期应支持：

- GitHub 仓库导入；
- 从 AI 工具复制“打包指令”；
- 下载/上传构建产物；
- 一键生成给 Codex/Claude Code/Cursor 的修复提示词。

## 6. 推荐产品路线

### v0.1.17：能力边界和部署体验修正

目标：让用户知道 DemoGo 能发布什么、不能发布什么，部署失败也能看懂原因。

建议范围：

- 项目类型检测增强；
- 明确区分：静态页面、已构建前端、可尝试构建源码、疑似表单项目、疑似后端项目、疑似 SSR 项目；
- 前端展示“支持情况”和“风险提示”；
- 对本地 `/api/*` 调用给出明确提示；
- 对 Next.js、Nuxt、Express、FastAPI 等给出暂不支持提示；
- JSON 解析错误兜底，避免展示 `Unexpected token '<'`；
- 部署超时提示改成人话；
- 生成给 AI 编程工具的修复指令；
- 对源码构建能力标注“实验支持”。

是否要在 v0.1.17 直接做完整异步部署：建议先做设计和最小化结构，不建议一次性大改。

### v0.1.18：DemoGo 表单托管第一版

目标：解决 AI 生成页面最常见的真实业务闭环。

建议范围：

- 创建表单；
- 字段配置；
- 表单提交 API；
- Demo 绑定表单；
- 提交记录保存；
- 用户端查看提交记录；
- 复制给 AI 编程工具的接入文案；
- 基础防刷；
- 套餐限制：Free 不开放或低额度，Lite/Pro 开放不同额度。

这是 DemoGo 从“能打开页面”到“能收集真实业务数据”的关键版本。

### v0.1.19：发布记录和数据记录产品化

目标：让用户知道额度为什么扣减，运营人员知道用户在干什么。

建议范围：

- 用户端发布记录；
- 每次发布/更新的状态、时间、项目、消耗额度；
- 表单提交记录导出；
- 项目维度数据汇总；
- 管理后台查看失败原因分布；
- 管理后台查看项目类型分布。

### v0.2.0：异步部署任务

目标：解决构建超时和发布流程稳定性。

建议范围：

- `deployment_jobs` 表；
- 上传后返回任务 ID；
- 后台执行检测、解压、构建、发布；
- 前端轮询状态；
- 部署日志结构化；
- 支持取消/重试；
- 失败后保留日志和修复建议。

### v0.2.x：Docker 构建隔离

目标：为源码构建和未来后端托管打基础。

建议范围：

- 临时构建容器；
- CPU/内存/时间限制；
- 禁止危险文件；
- 构建缓存策略；
- 网络访问策略；
- 构建产物提取；
- 容器销毁。

### v0.3.x：Node 后端实验支持

目标：支持最常见的全栈 AI 编程项目。

建议范围：

- 检测 `server.js`、`app.js`、`scripts.start`；
- 容器运行 Node 服务；
- 自动分配端口；
- `/api/*` 路由到容器；
- 日志查看；
- 健康检查；
- 自动停止过期 Demo；
- 环境变量配置。

### v0.4.x：数据库和完整应用托管

目标：从 Demo 发布平台进入轻量应用托管。

可选路线：

- SQLite per project；
- 共享 PostgreSQL/MySQL schema；
- 用户自带数据库连接；
- Supabase 集成；
- DemoGo 托管数据库。

建议不要太早做，因为一旦承诺数据库托管，就进入更重的运维和安全责任。

## 7. DemoGo 应该优先服务的用户场景

短期最值得服务的是：

1. 用 AI 做了一个宣传页，想马上发给客户看；
2. 用 AI 做了一个报名/预约/试用申请页，想收集真实名单；
3. 用 AI 做了一个产品原型，想让领导、客户、伙伴点击试用；
4. 用 AI 做了一个前端工具或小游戏，想快速分享；
5. 用 AI 做了一个 SaaS/CRM 前端界面，想先演示流程。

短期不宜承诺的是：

1. 完整后端应用上线；
2. 真实支付；
3. 高并发生产环境；
4. 复杂数据库应用；
5. 企业级权限体系；
6. 长期正式托管。

## 8. 首页和产品表达建议

DemoGo 首页应该避免把自己讲成“托管平台”或“云平台”，因为这会让用户期待完整 Render/Railway 能力。

更准确的表达是：

> 让 AI 做出来的产品，马上变成一个可试用、可分享、可收集反馈的链接。

核心卖点应围绕：

- 不用买域名；
- 不用备案；
- 不用配置服务器；
- 不用懂部署；
- 国内访问更稳定；
- 上传后生成可试用链接；
- 可以给客户、领导、伙伴、用户测试；
- 后续支持报名、预约、反馈数据收集。

需要明确边界：

- 当前主要支持静态页面和前端构建产物；
- 表单数据托管正在逐步开放；
- 完整后端、数据库、支付和 SSR 将作为后续能力。

## 9. 给 AI 编程工具的反向要求

DemoGo 可以逐步形成一套“AI 编程工具友好规范”，让用户把下面的要求丢给 Codex、Claude Code、Cursor、OpenCode 或 Hammers Agent：

```text
请把这个项目整理成适合 DemoGo 发布的版本：
1. 如果是静态页面，请确保根目录包含 index.html。
2. 如果是 React/Vue/Vite 项目，请确保 package.json 里有 npm run build，并能生成 dist/index.html 或 build/index.html。
3. 不要把 .env、密钥、node_modules、缓存目录打包进去。
4. 如果页面里有报名/预约/反馈表单，请把提交接口改成 DemoGo 提供的表单托管 API。
5. 如果项目依赖后端、数据库、支付或 API Key，请在 README 中明确说明这些功能当前不能通过纯静态发布直接运行。
```

这会让 DemoGo 不只是接收上传，而是成为 AI 编程产物的“发布规范制定者”。

## 10. 需要继续补充的真实调研

为了避免只凭平台宣传判断，建议后续建立一个样本库。

每个样本记录：

- 来源工具：Codex / Claude Code / Cursor / OpenCode / Hammers Agent / Bolt / Lovable / v0 / Replit；
- 项目类型；
- 文件结构；
- 是否有 `package.json`；
- 是否有 `scripts.build`；
- 是否有后端入口；
- 是否有数据库；
- 是否有表单；
- 是否有本地 API；
- 是否有 `.env`；
- DemoGo 是否能发布；
- 失败原因；
- 用户真实期望。

建议至少收集 50 个样本后，再决定 v0.3 以后后端能力的具体优先级。
