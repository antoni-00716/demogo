# DemoGo v0.1.9 版本计划：试用运营增强版

更新时间：2026-05-11

## 1. 版本定位

v0.1.9 不做单点小功能发布，而是作为一个完整的小版本。

目标是解决 v0.1.8 自测和真实试用前暴露出的运营问题：

- 管理后台不能手动把用户从免费版升级到付费版；
- 用户反馈只有“最近反馈”，缺少完整反馈列表和处理状态；
- Demo 发布后缺少可复制的转发文案，链接传播时没有自然带出 DemoGo 品牌；
- 需要对照 GitHub 上成熟开源产品，分析 DemoGo 在代码组织、架构、工程化上的差距；
- 管理员缺少对试用用户、问题反馈、发布额度的基本运营能力。

一句话定位：

> v0.1.9 是 DemoGo 的“试用运营增强版”，让管理员能支持第一批真实用户试用。

## 2. P0 必做功能

### A. 管理后台手动调整套餐

业务意义：

- 先不接在线支付，也能人工给用户开通 Lite / Pro；
- 适合早期内测、朋友试用、付费意向客户人工开通；
- 解决早期测试和试用过程中需要人工支持用户额度的问题。

功能范围：

- 管理后台用户列表显示当前套餐；
- 管理员可把用户从 `free` 调整为 `lite` 或 `pro`；
- 也支持从付费套餐调回 `free`；
- 调整后立即影响在线 Demo 数量、发布次数、保留天数等额度；
- 记录审计日志，包含管理员操作、原套餐、新套餐、用户邮箱、时间。

暂不做：

- 在线支付；
- 订单流水；
- 自动到期扣费；
- 发票、退款；
- 复杂会员权益系统。

### B. 参考成熟开源产品做架构差距分析和低风险改造

业务意义：

- DemoGo 不能只按当前 MVP 惯性继续加功能；
- 需要参考 Vercel / Netlify 类产品和 GitHub 上成熟开源部署平台的代码组织方式；
- 重点不是照搬，而是找出 DemoGo 当前架构的短板，并把该改的部分落实到 v0.1.9。

分析对象建议：

- Vercel：重点参考产品形态、CLI、部署抽象和项目结构，不假设其完整平台代码开源；
- Coolify：参考自托管 PaaS 的应用、部署、数据库、服务编排；
- Dokploy：参考现代开源部署平台的项目组织和服务边界；
- CapRover：参考轻量 PaaS 的部署模型和运维边界；
- 其他可补充：Railway、Netlify 的公开资料，作为产品架构参考。

输出物：

- DemoGo 当前代码结构复盘；
- 与成熟项目的架构差距表；
- 哪些差距现在必须补；
- 哪些差距等到 v0.2 / v0.3 再补；
- 是否需要从单文件后端逐步拆成模块化单体；
- 是否需要引入任务队列、构建 Worker、日志系统、权限模型、对象存储等。

v0.1.9 应落地的架构改造：

- 拆出 `config.js`，统一配置和版本号；
- 拆出鉴权/管理员认证中间件；
- 拆出套餐额度服务；
- 拆出反馈服务；
- 拆出审计日志服务；
- 管理后台 API 分组；
- 增加固定 smoke test，不再只依赖临时手工脚本。

暂不做：

- 大规模重构；
- 直接引入微服务；
- 直接照搬其他项目技术栈。

### C. 管理后台反馈列表

当前问题：

- v0.1.8 只有“最近反馈”；
- 运营上无法系统处理问题；
- 反馈多了以后无法搜索、筛选、标记状态。

功能范围：

- 管理后台新增“反馈列表”模块；
- 显示反馈类型、用户邮箱、关联 Demo、内容、状态、提交时间；
- 支持按关键词搜索；
- 支持按类型筛选；
- 支持按状态筛选：待处理、处理中、已处理、已关闭；
- 支持管理员修改反馈状态；
- 最近反馈仍保留在右侧概览。

### D. 管理后台用户搜索与套餐筛选增强

功能范围：

- 用户列表支持按邮箱搜索；
- 用户列表支持按套餐筛选；
- 用户列表显示用户的 Demo 数、在线 Demo 数、最近注册/活跃时间；
- 方便管理员判断哪些用户需要人工支持或升级。

### E. Demo 发布后的转发文案

业务意义：

- DemoGo 是 Demo 发布平台，链接生成后，用户天然需要把链接发给试用对象；
- 平台应提供一段可复制的转发话术，降低用户分享成本；
- 文案中适度带出 DemoGo，有利于自然传播和裂变；
- 每一个用户 Demo 链接都可以成为 DemoGo 的获客入口。

功能范围：

- 发布成功后展示“复制转发文案”；
- Demo 列表中每个已发布 Demo 提供“复制文案”按钮；
- 文案包含 Demo 名称、访问链接、试用提示和 DemoGo 品牌说明；
- 文案语气要自然，不能像硬广告；
- 如果检测报告发现表单/API 风险，文案不承诺数据提交一定可用；
- 后续可以支持不同场景模板：活动报名、产品演示、客户预览、内部评审。

推荐默认文案：

```text
我用 DemoGo 生成了一个可在线打开的 Demo 试用链接：

【Demo 名称】
访问链接：【Demo 链接】

你可以直接打开体验页面效果。如果有打不开、显示异常或需要调整的地方，请反馈给我。

这个链接由 DemoGo 生成。DemoGo 用来把 AI 工具生成的网页/产品 Demo 快速发布成可分享的试用链接，方便演示、测试和收集反馈。
```

暂不做：

- 多渠道自动分享；
- 二维码海报；
- 微信卡片；
- 邀请返利系统。

## 3. P1 建议功能

### F. 发布额度说明与耗尽提示优化

功能范围：

- 用户端显示当前套餐、在线 Demo 数、每月发布次数；
- 发布额度快用完时提示；
- 额度耗尽时提示“可联系管理员开通 Lite/Pro”；
- 管理后台调整套餐后，用户端刷新即可看到新额度。

### G. 管理后台基础运营备注

功能范围：

- 管理员可给用户添加一条内部备注；
- 例如“朋友试用”“意向客户”“已人工开通 Lite”；
- 备注只在管理后台显示。

是否纳入 v0.1.9 视开发量决定，不作为第一优先级。

## 4. 不纳入 v0.1.9

- 在线支付；
- 正式订单系统；
- 表单托管正式功能；
- 大模型 AI；
- Docker / 后端项目托管；
- 复杂权限系统；
- 微服务拆分；
- 大规模前端重构。

## 5. 数据库影响

预计需要最小化改动。

优先复用现有 `users.plan` 字段：

- `free`
- `lite`
- `pro`

新增或补充：

- 用户套餐调整写入 `audit_logs`；
- 反馈状态继续使用 `feedback.status`；
- 如做运营备注，可先写入用户 `metadata_json`，不新增表。

## 6. 验收标准

v0.1.9 完成后必须满足：

- 管理员能把免费用户手动升级为 Lite / Pro；
- 用户升级后额度立即生效；
- 已形成 DemoGo 与成熟开源部署产品的架构差距分析；
- 已完成 v0.1.9 范围内的低风险模块化改造；
- 管理员能查看完整反馈列表；
- 管理员能筛选和处理反馈；
- 用户发布 Demo 后能复制一段自然的转发文案；
- 转发文案能带出 DemoGo 品牌，但不夸大平台能力；
- v0.1.8 已有注册、登录、上传、检测、发布、下线、恢复、删除不被破坏；
- 发布前完成代码自审和自测，再交付部署。
