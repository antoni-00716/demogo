# DemoGo v0.1.11 版本计划：用户试用闭环版

更新时间：2026-05-12

## 1. 版本定位

v0.1.11 的目标不是继续堆新功能，而是把 DemoGo 的早期试用流程收敛清楚。

一句话定位：

> 让用户能发布、能理解额度、能查看发布记录、能申请升级；让管理员能处理升级线索和反馈问题。

当前 DemoGo 已经具备发布能力，但用户端和后台还有几个断点：

- 用户不知道套餐差异；
- 用户不知道发布次数为什么减少；
- 用户申请升级和后台开通流程还不够唯一；
- 后台有重复入口，容易混乱；
- 复制转发文案的体验应该更直接；
- “订单记录”在当前阶段没有真实业务意义。

v0.1.11 要解决的是“试用闭环”，不是支付闭环。

## 2. 产品目标

### 用户侧目标

用户进入控制台后，应该清楚知道：

- 当前自己是什么套餐；
- 每个套餐有什么权益；
- 自己还剩多少发布/更新额度；
- 哪些发布/更新消耗了额度；
- 如何申请升级；
- 申请升级后处于什么状态；
- Demo 链接如何一键转发给别人。

### 运营侧目标

管理员进入后台后，应该清楚知道：

- 谁提交了升级申请；
- 谁的问题反馈待处理；
- 哪些用户已经真实发布 Demo；
- 哪些用户有升级意向；
- 哪些操作已经被处理并留下记录。

### 技术侧目标

本版本继续坚持小步架构演进：

- 不做大重构；
- 不做微服务；
- 新增能力进入清晰的数据结构和服务边界；
- 用 smoke test 覆盖核心流程；
- 避免继续扩大 `server.js` 的历史债务。

## 3. 用户端改造

### A. 页面结构调整

建议用户端主结构调整为：

1. 我的 Demo；
2. 上传发布；
3. 套餐与额度；
4. 本月发布记录；
5. 反馈问题。

删除或改造：

- 删除“订单记录”；
- 当前阶段不展示虚假的订单信息；
- “套餐额度”改为“套餐与额度”。

### B. 套餐卡片化

用卡片展示三个套餐：

| 套餐 | 在线 Demo | 发布/更新次数 | Demo 保留 | 适合场景 |
|---|---:|---:|---:|---|
| 免费体验 | 1 个 | 3 次/月 | 7 天 | 初次试用 |
| 轻量版 | 3 个 | 20 次/月 | 30 天 | 客户演示、小项目验证 |
| Pro | 10 个 | 60 次/月 | 30 天 | 多项目持续试用 |

交互规则：

- 当前套餐明确标记“当前套餐”；
- 免费版用户可申请轻量版或 Pro；
- 轻量版用户可申请 Pro；
- Pro 用户显示“已是最高套餐”；
- 用户不能自己降级；
- 当前版本不做套餐有效期。

### C. 升级申请状态

用户端显示最近一次升级申请状态：

- 待处理；
- 已开通；
- 已拒绝。

如果存在待处理申请：

- 禁止重复申请；
- 显示“管理员处理中”；
- 允许用户继续使用现有套餐能力。

### D. 本月发布记录

新增“本月发布记录”模块，用来解释额度扣减。

展示字段：

| 字段 | 说明 |
|---|---|
| 时间 | 发布或更新时间 |
| Demo | Demo 名称或 slug |
| 操作 | 发布 / 更新 |
| 版本 | V1 / V2 |
| 额度影响 | 计入本月发布次数 |

计数规则：

- 成功发布计入；
- 成功更新计入；
- 检测失败不计入；
- 发布失败暂不计入；
- 下线、恢复、删除不计入。

实现优先级：

- 优先复用现有 `deployEvents` / `demo_versions`；
- 如现有数据足够，先不新增表；
- 后续 Worker 化后再抽象为 `deployment_jobs`。

### E. 复制转发文案

目标体验：

> 点击“复制文案”后，直接复制成功，按钮显示“已复制”。

交互规则：

- 从 Demo 列表点击复制，不跳到上传区域；
- 不默认展示大文本框；
- 不使用浏览器原生 prompt；
- 优先使用浏览器剪贴板 API；
- 失败时用隐藏 textarea + `execCommand("copy")` 兜底；
- 仍失败时才显示页面内备用文本框。

## 4. 管理后台改造

### A. 升级申请作为唯一开通入口

后台套餐开通只走“升级申请”。

改造内容：

- 保留“升级申请”模块；
- 管理员在这里点击“开通”；
- 开通后自动更新用户套餐；
- 拒绝时页面内填写原因；
- 写入审计日志。

### B. 用户列表只读

后台可以保留用户列表，但改为只读。

展示：

- 用户邮箱；
- 当前套餐；
- Demo 总数；
- 在线 Demo 数；
- 注册时间；
- 最近活跃时间，若现有数据不足可后置。

不再展示：

- 套餐下拉；
- 手动保存套餐按钮。

原因：

- 避免“用户与套餐”和“升级申请”两个入口重复改套餐；
- 保持运营流程唯一；
- 降低误操作风险。

### C. 后台运营待办

后台首页应突出：

- 待处理升级申请数；
- 待处理反馈数；
- 最近发布 / 更新记录；
- 可疑风险或失败提示，若暂时数据不足可后置。

## 5. 后端与数据设计

### A. 已有能力复用

v0.1.10 已有：

- `plan_upgrade_requests`；
- `plan-request-service.js`；
- 升级申请 API；
- 管理后台处理 API；
- 审计日志。

v0.1.11 优先复用，不重复建表。

### B. 新增 API 建议

新增用户发布记录接口：

```text
GET /api/deploy-events
```

返回当前用户本月发布/更新记录。

建议返回：

```json
{
  "events": [
    {
      "demoId": "xxx",
      "demoName": "活动报名页",
      "slug": "event-demo",
      "type": "create",
      "typeLabel": "发布",
      "version": 1,
      "counted": true,
      "createdAt": "2026-05-12T13:20:00.000Z"
    }
  ]
}
```

### C. 套餐规则

当前版本不新增套餐有效期。

保留现有规则：

- 套餐决定 Demo 在线数量；
- 套餐决定本月发布/更新次数；
- 套餐决定新发布或更新 Demo 的保留天数。

不做：

- 套餐到期；
- 自动降级；
- 订单续费；
- 支付状态。

## 6. 架构演进项

按照“每个版本偿还一小块架构债”的原则，v0.1.11 建议做：

- 新增 `deploy-event-service.js`，封装发布记录计算；
- 前端套餐权益集中成一个配置对象，避免散落在 DOM 和逻辑里；
- 后端发布记录接口进入 smoke test；
- 管理后台套餐改动入口收敛到升级申请；
- 为后续 `deployment_jobs` 表预留概念，但本版本不建。

## 7. 不纳入本版本

- 在线支付；
- 正式订单系统；
- 套餐有效期；
- 自动降级；
- 表单托管正式功能；
- 大模型 AI；
- Docker / 后端托管；
- 完整微服务拆分；
- 对象存储和 CDN；
- OpenAPI / CLI。

## 8. 验收标准

用户端：

- 套餐以卡片展示；
- 当前套餐标识清楚；
- 可以申请合适的升级套餐；
- Pro 用户不能申请更高套餐；
- 订单记录不再出现；
- 本月发布记录能解释额度扣减；
- 复制文案点击后优先直接复制，不跳转上传区。

管理后台：

- 升级申请是唯一开通入口；
- 用户列表不再提供套餐下拉修改；
- 开通申请后用户套餐生效；
- 拒绝申请可页面内填写原因；
- 待处理升级申请数清楚可见。

技术：

- 不破坏 v0.1.10 线上能力；
- 不调整免费版额度；
- 不引入支付或订单系统；
- `npm run check` 通过；
- `npm run test:smoke` 通过；
- 部署前提供备份、验证和回滚步骤。
