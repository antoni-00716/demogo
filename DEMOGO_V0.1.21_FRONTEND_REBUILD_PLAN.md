# DemoGo v0.1.21 前端整体重构方案

## 版本定位

v0.1.21 定位为：**三端前端重构 + AI 发布入口基础版**。

这一版不是小修小改，而是把首页、用户端、管理后台作为一个完整产品重新设计。目标是让用户第一次打开 DemoGo 时产生三个感受：

1. 这个产品看起来专业、清爽、可信。
2. 我能马上理解它帮我解决什么问题。
3. 我愿意上传一个 AI 做出来的项目试一下。

同时保留原计划里的 AI 发布方向，为后续 `npx demogo deploy`、MCP、Codex 插件打基础。

## 对标结论

本次参考 Vercel、Netlify、Railway、Render、Zeabur，但不照搬。

- Vercel 强调体验、速度和设计感，产品表达非常克制，强调“构建和部署应该足够简单”。
- Netlify 已经把 AI agents、部署预览、表单、后端能力统一放进平台表达里，值得参考其“AI 工具 + 平台处理后续”的叙事。
- Railway 的核心表达是把项目来源交给平台，平台处理构建、发布和运行，CLI/API 是工作流的一部分。
- Render 对静态网站的表达清楚：连接项目、设置构建、生成唯一访问地址。
- Zeabur 强调一键部署、自动识别项目类型、快速生成域名，和 DemoGo 的 MVP 场景高度相关。

参考来源：

- Vercel About: https://vercel.com/about
- Netlify Platform: https://www.netlify.com/platform/
- Railway Platform: https://docs.railway.com/platform
- Railway CLI: https://docs.railway.com/cli
- Render Static Sites: https://render.com/docs/static-sites/
- Zeabur Introduction: https://zeabur.com/docs/en-US

DemoGo 的差异化不是“更强的云平台”，而是：

> 面向非技术用户，把 AI 做出来的页面快速变成国内可访问、可转发、可收集报名/留言的试用链接。

## 当前主要问题

### 首页

- 第一屏仍然偏技术产品表达，没有形成足够强的“试用链接平台”心智。
- “免费开始”和线上“内测发布码”存在预期冲突。
- 部分词汇仍像内部研发语言，例如 Demo、发布、部署、表单托管、Agent。
- 视觉上虽有提升，但还没有形成国际平台级别的高级感和统一秩序。

### 用户端

- 工作台像管理后台，不像“我的试用项目工作台”。
- 项目详情内容顺序偏技术，用户最想看的“链接、转发、报名/留言记录”没有被优先突出。
- 创建项目页仍有工程感，用户不知道自己应该做什么、下一步会得到什么。
- “表单托管、提交地址、备用接入指令”对非技术用户仍然难理解。
- “AI 发布、发布密钥”偏技术，需要改成用户能理解的表达。

### 管理后台

- 后台已经有运营结构，但还不够像“今日运营工作台”。
- 运营人员最关心的是待处理事项，不是先看一堆指标。
- 表单记录更像技术记录，后续需要变成业务线索记录。
- 项目风险、发布失败、用户反馈、升级申请之间缺少明确处理优先级。

## 产品语言体系

全站统一把内部词汇改成用户语言。

| 当前词汇 | v0.1.21 建议词汇 |
|---|---|
| Demo | 试用项目 |
| 发布 | 生成试用链接 / 上线试用 |
| 部署 | 生成试用链接 |
| 表单托管 | 报名/留言收集 |
| 提交地址 | 收集入口 |
| Agent 发布 | 让 AI 帮我发布 |
| 发布密钥 | AI 发布口令 |
| API | 后台功能 |
| 后端 | 登录、订单、支付、数据库等后台功能 |
| 体检 / Inspection | 发布前检查 |
| 风险 | 需要注意 |
| 部署失败 | 生成链接失败 |

原则：

- 首页不直接强调 React、Vue、Vite、API、Agent。
- 检测结果里可以出现技术事实，但必须先给用户语言解释。
- 不把“不支持”包装成“实验能力”。支持就是支持，不支持就是不支持。

## 三端信息架构

### 首页

首页目标：让用户在 10 秒内理解 DemoGo。

建议结构：

1. 顶部导航
   - Logo
   - 能做什么
   - 使用流程
   - 适合场景
   - 套餐
   - 登录 / 开始试用

2. 第一屏
   - 主标题：AI 做好的页面，一键变成可试用链接
   - 副标题：上传 AI 工具生成的项目包，DemoGo 自动检查页面、生成访问链接，还能收集报名、预约和留言。
   - 主按钮：开始生成试用链接
   - 次按钮：查看支持范围
   - 右侧展示真实产品界面预览，而不是抽象流程卡片。

3. 真实问题
   - 页面做好了，但客户打不开。
   - 不会买服务器、配域名、做备案。
   - 报名表看起来能填，但数据收不到。
   - 发给别人试用时，不知道哪些功能其实不能用。

4. 使用流程
   - 上传项目包
   - 发布前检查
   - 生成试用链接
   - 分享给别人
   - 查看报名/留言

5. 适合场景
   - 活动报名页
   - 培训招生页
   - 产品原型
   - 个人作品集
   - 课程作业
   - 内部工具演示

6. 当前支持 / 暂不支持
   - 支持：普通网页、AI 生成页面、静态项目、能生成网页版本的前端源码、基础报名/留言收集。
   - 不支持：长期运行的后台服务、数据库、登录系统、订单支付、WebSocket、SSR 运行时。

7. 套餐与内测说明
   - 如果继续内测：明确“当前为邀请内测，需要发布码”。
   - 如果开放试用：去掉发布码，用额度控制。

8. 底部
   - 产品边界
   - 隐私与内容规则
   - 联系方式

### 用户端

用户端目标：让用户围绕“试用项目”完成闭环。

左侧导航建议：

- 工作台
- 我的试用项目
- 生成新链接
- 报名/留言记录
- 让 AI 帮我发布
- 套餐与额度
- 问题反馈

页面结构：

1. 工作台
   - 今日重点：正在试用的项目、收到的报名/留言、本月剩余额度。
   - 最近项目：复制链接、复制转发文案、查看详情。
   - 下一步建议：没有项目时引导上传；有项目时引导分享和查看反馈。

2. 我的试用项目
   - 使用列表为主，不做卡片堆叠。
   - 每行展示：项目名称、状态、访问链接、报名/留言数量、最近更新、操作。
   - 操作：复制链接、转发文案、查看详情、更新、下线。

3. 项目详情抽屉
   - 第一块：访问链接和核心操作。
   - 第二块：报名/留言记录。
   - 第三块：发布前检查结果。
   - 第四块：发布历史。
   - 删除、下线等危险操作放到底部。

4. 生成新链接
   - 步骤式布局：
     1. 上传项目包
     2. 看看能不能试用
     3. 生成试用链接
     4. 分享给别人
   - 如果仍保留内测发布码，必须在页首解释清楚。

5. 报名/留言记录
   - 单独入口，方便用户不进入项目详情也能查看。
   - 展示：项目、姓名/联系方式、提交时间、留言内容。
   - 后续可加导出 CSV、标记已处理。

6. 让 AI 帮我发布
   - 不再主打 Agent、Token。
   - 文案改成：复制这段指令给 Codex、Cursor、Claude Code，它会帮你打包当前项目并生成试用链接。
   - “AI 发布口令”只在必要处出现，并解释“相当于让 AI 代表你发布项目的授权码”。

### 管理后台

管理后台目标：让运营人员知道今天该处理什么。

左侧导航建议：

- 今日待处理
- 用户
- 试用项目
- 报名/留言
- 升级申请
- 用户问题
- 系统设置

页面结构：

1. 今日待处理
   - 待处理升级申请
   - 生成链接失败的项目
   - 有新反馈未处理
   - 新增报名/留言
   - 接近额度上限的用户

2. 用户
   - 用户邮箱、套餐、在线项目、累计项目、最近活跃。
   - 详情抽屉展示用户项目、套餐、升级申请、反馈。

3. 试用项目
   - 项目、用户、状态、访问量、报名/留言数量、需要注意事项。
   - 详情抽屉展示链接、检查结果、发布历史、运营操作。

4. 报名/留言
   - 项目、提交人、联系方式、提交内容、提交时间。
   - 后续支持标记已跟进、导出。

5. 升级申请
   - 优先展示待处理。
   - 处理动作在详情抽屉中完成。

6. 用户问题
   - 待处理、处理中、已处理。
   - 保留处理记录，避免用户反馈丢失。

## 视觉系统

### 设计方向

关键词：清爽、可信、轻量、高级、面向非技术用户。

不走传统后台大蓝灰风格，也不做花哨营销页。DemoGo 应该像一个“轻量但可靠的 AI 试用链接工作台”。

### 色彩

- 主色：DemoGo 绿色，用于按钮、状态、重点数字。
- 背景：灰白、淡绿、白色为主。
- 文本：深墨色主文本、灰蓝辅助文本。
- 警告：柔和橙色。
- 危险：克制红色。

要求：

- 绿色背景上不要放黑字，优先用白字或深绿色低饱和背景。
- 不做大面积单一绿色。
- 不用大面积紫蓝渐变、深蓝暗黑、咖啡色等不符合当前品牌的主题。

### 布局

- 首页第一屏允许更有视觉冲击。
- 用户端和后台以效率为主，不做营销式大卡片堆叠。
- 列表优先，卡片只用于摘要、空状态、关键操作。
- 同一行卡片必须等高、对齐。
- 不做横向拖拽表格。
- 不出现单字换行。
- 按钮组需要固定间距和宽度规则。

### 组件

需要统一的组件规范：

- Button
- Badge
- Panel
- Drawer
- Table/List
- Empty State
- Toast
- Stepper
- Upload Zone
- Form Record Row
- Plan Card

## 发布码问题

这是 v0.1.21 必须处理的问题。

当前线上真实测试发现：新用户可以注册、可以检测项目，但发布时会被“内测发布码不正确”拦住。这和首页“免费开始”冲突。

建议采用方案 A：

### 方案 A：取消普通用户发布码

- 用户注册后可直接按 Free 套餐额度发布。
- 通过在线项目数、发布次数、文件大小控制风险。
- 后台保留用户下线、删除、反馈、升级处理能力。

优点：

- 用户体验最顺。
- 首页“免费开始”成立。
- 有利于真实试用和转化。

风险：

- 需要关注滥用和垃圾上传。
- 可通过额度、文件大小、危险文件拦截、后台下线来控制。

### 方案 B：继续邀请内测

- 首页、登录页、创建项目页都明确写“邀请内测，需要发布码”。
- 按钮改成“申请内测”或“进入内测发布”。

优点：

- 风险更低。

问题：

- 用户转化路径变长。
- 产品看起来不够开放。

建议：v0.1.21 采用方案 A，取消普通发布码。

## CLI / AI 发布入口

v0.1.21 保留原计划里的 AI 发布方向，但只做基础版。

目标：

- 用户可以在 AI 编程工具里发布当前项目。
- CLI 复用现有 `/api/agent/deploy`。
- 不重写后端发布逻辑。

第一版能力：

- 生成/重置 AI 发布口令。
- 用户端复制给 AI 工具的发布指令。
- CLI 初版命令：`npx demogo deploy`。
- CLI 做本地打包、上传、返回试用链接。

暂不做：

- MCP Server。
- Codex 插件完整市场化发布。
- 自动数据库。
- 后端项目托管。

## 验收标准

### 用户体验

- 用户 10 秒内能理解 DemoGo 是干什么的。
- 用户能知道自己应该上传什么。
- 用户知道上传后会得到什么。
- 用户知道哪些能力当前不支持。
- 注册后不会被未解释的发布码卡住。
- 创建项目流程没有技术化断点。
- 报名/留言记录能被用户自然找到。

### 运营体验

- 管理员打开后台能先看到待处理事项。
- 升级申请、用户反馈、表单记录、失败项目都有清晰入口。
- 详情信息通过抽屉展示，不在页面下方堆叠。
- 运营人员能判断哪些项目需要关注。

### 视觉体验

- 首页、用户端、后台像同一个系统。
- 页面清爽、有高级感，不像临时拼出来的后台。
- 无单字换行。
- 无按钮遮挡。
- 无横向拖拽表格。
- 同一行卡片等高对齐。
- 绿色背景文字清晰。
- 移动端不崩。

### 技术验证

- `server npm run check` 通过。
- `server npm run test:smoke` 通过。
- `web npm run lint` 通过。
- `web npm run build` 通过。
- 项目类型矩阵测试通过。
- 首页、登录页、用户端、管理后台本地和线上均可打开。
- v0.1.21 部署包内容正确。

## 风险与取舍

### 风险 1：重构范围较大

控制方式：

- 不改核心发布逻辑。
- 不重写后端数据结构。
- 前端按页面逐步替换。
- 每完成一端就截图验收。

### 风险 2：取消发布码带来滥用

控制方式：

- Free 套餐限制在线项目数、发布次数、文件大小。
- 保留危险文件拦截。
- 后台可下线和删除。
- 后续再加邮箱验证、频率限制、IP 限制。

### 风险 3：CLI 和前端重构同时做，版本偏大

控制方式：

- CLI 只做最小可用版。
- 优先保证三端体验。
- 如时间压力过大，CLI 可只完成本地包结构和基础发布命令，不做高级配置。

## 结论

v0.1.21 应该作为 DemoGo 从“能用的 MVP”走向“用户愿意试用的 MVP”的关键版本。

这一版最重要的不是增加更多复杂功能，而是把用户第一次接触产品的体验做好：

- 看起来专业。
- 讲得清楚。
- 上传顺畅。
- 结果明确。
- 后台能运营。
- AI 发布方向开始落地。
