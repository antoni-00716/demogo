## Executive Summary

**16 functions are duplicated** between `server.js` and `deployment-pipeline-service.js`. The two files have diverged — they share the same logical structure but differ in:
- Chinese user-facing message strings (slightly different phrasings)
- Inspection object property access paths (flat vs nested, e.g. `inspection.hostingMode` vs `inspection.analysis.hostingMode`)
- User-facing presentation data structures

Additionally, server.js **still holds local copies** of functions that deployment-pipeline-service has already migrated to shared libs (`slug-utils.js`, `deployment-steps.js`, `utils.js`).

## Key Risks & Notes

1. **Inspection property path divergence**: The two files use different paths for the same inspection data (flat vs nested under `analysis`/`presentation`/`entries`). T05 must pick ONE canonical schema and update both callers.
2. **`logger` reference**: server.js has `logger` imported globally. deployment-pipeline-service.js doesn't import logger at the top — some functions reference `logger` inline (line 1416 in pipeline file for `performUpdateDeployment`). This needs a logger import added.
3. **Circular dependency risk**: deployment-pipeline-service already uses lazy injection (`setFileHandlers`, `setBuildFunctions`) to avoid circular dependencies. Any refactoring must maintain or break the cycle properly.
4. **Message string divergence**: Both files have Chinese user-facing strings that differ. Decide which is canonical before T05.
