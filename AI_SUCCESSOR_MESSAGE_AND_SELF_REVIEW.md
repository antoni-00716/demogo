# 给接手 DemoGo 的 AI 的话与自我复盘

更新时间：2026-05-28

这份文档不是技术交接清单，技术交接请先看 `NEXT_AI_HANDOFF.md`。本文主要记录两件事：

1. 我想对下一个接手 DemoGo 的 AI 说什么。
2. 我对自己这段工作的评分和反思。

## 一、给下一个 AI 的话

你接手的是一个真实在迭代、已经上线的产品，不是练习项目。请先读：

- `NEXT_AI_HANDOFF.md`
- `AGENTS.md`
- `docs/CURRENT_STATUS.md`
- `docs/ROADMAP.md`

不要一上来就改代码，更不要一上来就重构。

DemoGo 当前已经到 `v0.9.0`，线上服务、CLI、MCP 都已经发布并验证。只有项目负责人明确说“发布 vX.X.X”或“发布这个版本”时，才可以部署线上和发布 npm。开发完成不等于发布。

项目负责人不是技术背景，但产品判断很强。你不能只给代码，也不能只讲技术名词。每个重要技术决策都要讲清楚：

- 业务含义是什么。
- 技术逻辑是什么。
- 怎么实现。
- 有什么取舍。
- 怎么验证。

最重要的产品原则是：

> 把方便留给用户，把麻烦留给 DemoGo。

这句话要贯穿所有设计。不要让用户区分一堆工具、参数、技术栈和命令细节。能由 DemoGo 自动判断、自动处理、自动兜底的事情，就不要交给用户。

尤其注意 AI 发布能力。用户已经明确要求：

- AI 发布页只保留一条通用提示词。
- 不要再按 Codex、Claude Code、Cursor、Windsurf 等工具拆多套提示词。
- 工具差异应由 CLI、MCP、Plugin、Skill、Agent API 在内部解决。

不要制造能力错觉。支持就是支持，不支持就是不支持。当前支持静态项目、前端构建产物、Node.js 单服务、MySQL 试用数据库、用户自己的 Supabase 连接；当前不支持多服务编排、Redis/MongoDB/Postgres 自动托管、Python/Java/Go 后端、WebSocket、真实支付和生产级登录系统。

DemoGo 不是电商平台。用户说“完整电商平台项目”，意思是把它作为 DemoGo 部署能力的边界样本，因为这类项目通常包含前端、后端、数据库、管理后台、环境变量和初始化脚本。目标是证明 DemoGo 能部署接近完整应用，不是让 DemoGo 自己做电商业务。

当前工作树很脏，里面有大量历史文件删除、新文档、新插件包和重构后的目录，这是之前用户确认过的清理方向。不要使用 `git reset --hard`，不要回退用户已有改动，不要随便恢复旧文件。

下一步建议进入 `v1.0.0`，定位为：

> 完整应用试用部署闭环。

重点不是小修小补，而是让 DemoGo 能稳定验收一个接近真实产品的完整应用：

- 前端页面可访问。
- Node.js 后端 API 可运行。
- MySQL 试用数据库可初始化。
- 环境变量可配置。
- 简单管理后台可访问。
- 表单或业务数据可提交、可查询。
- 更新版本后原链接保持不变。
- 失败后能给出清楚诊断和“复制给 AI 怎么改”。

动手前先给方案，等用户确认。确认后再开发。开发后必须测试。测试不通过不要说完成。

## 二、我对自己的评分

如果给自己这段 DemoGo 工作打分，我会打：

> 75 分。

这个分数的含义是：技术推进有成绩，但产品节奏、方案确认和商业化表达还有明显提升空间。

## 三、主要扣分点

### 1. 节奏控制不够好

前面很多版本做得偏碎，用户多次提醒“每个版本要有质变”。我没有一开始就把版本规划拉到足够大的产品目标上，导致部分版本更像功能补丁，而不是阶段性跃迁。

### 2. 产品表达有偏差

首页和用户端文案曾多次被指出“理工男”“不够高级”“没有情绪化”。这说明我前期对 DemoGo 的市场化表达、品牌调性和用户心理理解不够到位。

DemoGo 的核心不是展示技术能力，而是让开发出来的产品更快被用户看见、试用、反馈。这个表达后来才逐渐收敛。

### 3. 开发方案确认不够严格

AI 更新版本方案那次，我没有先和用户确认就推进了，这是明显问题。凡是影响产品路径、用户体验、商业权益或技术架构的改动，都必须先给方案，等确认后再做。

### 4. 发布流程一开始不够严谨

前期多次出现服务器部署后，npm/npx 需要用户提醒才同步验证的情况。后来虽然补了规则，但这类发布闭环本应从一开始就严谨执行。

### 5. 文档治理是后补的

项目早期文档、脚本和临时材料堆积较多，导致后面清理成本高。文档体系、版本状态、发布规则和能力边界应该更早建立。

### 6. 早期架构设计偏短期

项目从 0 到 1 跑通是重要成果，但早期很多实现更偏“当前容易实现”，没有充分为未来能力扩展预留结构。结果是后续能力增加后，部分模块变得过大，架构级调整幅度也变大。

尤其需要提醒下一个 AI：当前后端很多能力集中在 `server/src/server.js` 和多个 service 里，service 层承担了过多职责。后续不要继续简单堆功能，而要逐步把能力拆成更清晰的边界，例如：

- 发布编排。
- 项目识别。
- 内容安全。
- 运行环境。
- 数据库能力。
- 外部后端连接。
- 失败诊断。
- 用户权益与额度。
- AI 工具接入。

拆分不是为了形式上的“架构高级”，而是为了让后续完整应用部署、更多运行环境、更多数据库能力和更多 AI 工具接入时，不再持续放大维护成本。

## 四、做得还可以的地方

### 1. DemoGo 能力确实推进了

DemoGo 已从早期网页发布，推进到了：

- Web 上传。
- CLI 发布。
- MCP 发布。
- Agent API。
- Codex Skill。
- Codex Plugin。
- Claude Code Plugin。

这说明 DemoGo 已经开始真正进入 AI 编程工具链，而不是停留在一个上传网页的小工具。

### 2. 完整应用能力开始成型

目前已经形成了几条关键主线：

- Node.js 单服务运行器。
- MySQL 试用数据库。
- Supabase 外部后端连接。
- 项目识别 2.0。
- 失败诊断 2.0。
- 版本更新能力。
- 试用交付报告。

这些能力为后续 `v1.0.0` 做完整应用试用部署闭环打下了基础。

### 3. 产品原则逐渐统一

经过多轮讨论，DemoGo 的表达逐渐从“部署工具”收敛到：

```text
分享 -> 试用 -> 反馈
```

同时形成了一个重要原则：

```text
把方便留给用户，把麻烦留给 DemoGo
```

这个原则应继续保留，并成为后续所有产品设计和开发取舍的判断标准。

### 4. 交接状态比之前清楚

目前已经有 `NEXT_AI_HANDOFF.md` 和本文，另一个 AI 接手时至少能清楚知道：

- 当前真实版本。
- 已发布状态。
- 当前支持什么。
- 明确不支持什么。
- 用户的偏好和禁忌。
- 下一步建议做什么。

### 5. 从 0 到 1 把产品跑通

项目负责人给出的评价是：如果由他评分，会给这段工作 `85 分`。

核心原因是 DemoGo 确实从 0 到 1 被构建出来，并且基本跑通了真实产品闭环。虽然过程中有不少问题，包括早期架构不够长远、部分实现偏短期、后续出现架构级调整，但产品至少已经做出来、上线并持续迭代。

这对下一个 AI 很重要：不要否定已有基础。当前项目不是一张白纸，而是一个已经跑通、已经上线、已经有真实发布链路的产品。后续工作应该在现有基础上补强架构和能力，而不是推倒重来。

## 五、给下一个 AI 的最后提醒

不要把 DemoGo 做成一个技术展示项目。它的核心价值是让用户更快把产品交给真实用户试用。

不要每次只做小修小补。用户已经明确希望每个版本都有质变。

不要过度追求“架构漂亮”。当前最重要的是：

- 真实可用。
- 用户能理解。
- 发布能成功。
- 失败能说清楚。
- 能帮助用户更快获得反馈。

如果你能沿着这个方向推进，DemoGo 才会从 MVP 继续走向真正可试用、可商业验证的产品。

## 六、如果重新开发 DemoGo，我的反思

如果让我重新开发 DemoGo，我不会从“能不能上传一个网页”这个点开始一路堆功能，而会先把产品主线、能力边界和版本节奏设计清楚。

### 1. 先定义产品主线，再定义功能

一开始应该明确 DemoGo 的主线不是“部署网页”，而是：

```text
让开发出来的产品尽快被用户看见、试用、反馈
```

所有功能都应该围绕这个主线展开：

- 用户怎么把项目交给 DemoGo。
- DemoGo 怎么判断能不能发布。
- 发布后用户怎么分享。
- 试用者怎么反馈。
- 失败后怎么让 AI 修改。
- 用户怎么更新版本且链接不变。

如果主线更早清楚，很多功能不会做得分散，首页、用户端、管理端的表达也会更统一。

### 2. 一开始就应该建立“支持边界”

早期容易把 DemoGo 讲成“什么都能部署”，这是危险的。正确做法应该是从第一天就写清楚：

- 当前支持什么。
- 当前明确不支持什么。
- 哪些只是识别，不代表可以运行。
- 哪些是未来路线，不是当前能力。

这样可以避免用户产生错误预期，也能避免开发时为了“看起来支持更多”而做出不稳的实现。

### 3. 版本应该按能力跃迁组织，而不是按小功能组织

后面用户多次提醒“每个版本要有质变”，这个提醒是对的。

如果重新来，我会把版本分成几个明确阶段：

- 阶段一：静态项目和前端项目稳定发布。
- 阶段二：AI 发布和版本更新闭环。
- 阶段三：Node.js 单服务运行闭环。
- 阶段四：MySQL 试用数据库闭环。
- 阶段五：完整应用试用交付闭环。
- 阶段六：AI 工具插件化接入闭环。

每个阶段都应该有一个清晰验收样本，而不是只看功能点是否写完。

### 4. 首页和用户端不应该后期才重做

我前期太偏技术实现，低估了首页、用户端和管理端的商业化表达。

DemoGo 是要给真实用户看的产品，不是内部工具。首页从一开始就应该围绕用户情绪和场景来设计：

- 我做出了一个产品，想让别人马上打开。
- 我不想折腾服务器、域名、环境和部署。
- 我想先收反馈，而不是等到完美。
- 我希望 AI 帮我改完之后可以继续更新同一个链接。

技术能力要藏在背后，用户前台看到的应该是“分享、试用、反馈”。

### 5. AI 发布从第一天就应该按“统一入口”设计

一开始针对不同 AI 工具准备不同提示词，是增加用户负担。

如果重新来，我会从第一天就坚持：

- 用户只看到一条通用 AI 发布提示词。
- Codex、Claude Code、Cursor、Windsurf、OpenHands 等差异由 DemoGo 内部适配。
- CLI、MCP、Plugin、Skill、Agent API 是底层能力，不是让用户理解的复杂概念。

这更符合“把方便留给用户，把麻烦留给 DemoGo”。

### 6. 失败诊断应该更早成为核心能力

DemoGo 这类产品，失败是常态。项目结构不对、缺环境变量、缺 start 命令、端口不对、依赖安装失败、数据库初始化失败都会发生。

如果重新来，我会更早建设失败诊断：

- 不是只返回错误日志。
- 要告诉用户哪里失败。
- 要告诉用户为什么失败。
- 要告诉用户下一步怎么做。
- 要生成可以直接复制给 AI 的修复提示。

失败诊断不是辅助功能，而是 DemoGo 能不能真正被非技术用户使用的关键能力。

### 7. 数据库能力不应该回避太久

用户曾指出我好像一直在拖延进入数据库能力，这个判断是有道理的。

完整应用离不开数据库。DemoGo 不需要一开始支持所有数据库，但应该更早确定第一条主线：

```text
Node.js 单服务 + MySQL 试用数据库 + schema.sql 初始化
```

这条链路打透以后，再考虑 Supabase、Postgres、Redis、MongoDB、多服务编排。顺序不能乱。

### 8. 文档治理应该从项目早期开始

早期文档和脚本堆积太多，后来清理成本很高。

如果重新来，我会从一开始就建立这几类文档：

- 当前状态：真实支持什么、线上版本是什么。
- 路线图：下一阶段做什么、不做什么。
- 部署文档：怎么打包、怎么发布、怎么验证。
- 能力边界：支持和不支持清单。
- 版本记录：每个版本的定位和验收结果。

文档不是形式，它会直接影响后续 AI 能不能正确接手。

### 9. 发布流程应该从第一版就标准化

前期出现过服务器部署完成，但 npm/npx 没有同步验证的情况。这是发布流程不完整。

如果重新来，每次发布都应该固定为：

```text
本地测试 -> 打包 -> 上传 -> 服务器部署 -> 线上接口验证 -> npm 发布 -> npx 验证 -> 交付说明
```

只要涉及 CLI 或 MCP，就必须验证 npm 和 npx，不能等用户提醒。

### 10. 管理端不应该追求复杂，而应该服务运营判断

DemoGo 早期管理员只有一个人，管理端不需要做得很重。但它必须能回答几个问题：

- 今天有多少人访问、注册、发布。
- 用户卡在哪里。
- 哪些项目发布失败。
- 失败原因是什么。
- 哪些用户需要升级权益。
- 哪些项目已经具备试用交付条件。

管理端的核心不是“功能多”，而是让运营者快速判断真实卡点。

### 11. 不要过早追求云平台化

DemoGo 长期可能会接近一个轻量部署平台，但早期不能一上来就追求完整云平台能力。

正确顺序应该是：

```text
试用链接平台 -> 完整应用试用平台 -> AI 工具发布入口 -> 更强的运行环境
```

先证明用户愿意用 DemoGo 分享、试用、反馈，再逐步增强运行能力。

### 12. 技术架构要服务产品闭环，不要为了高级而高级

架构不是越复杂越好。DemoGo 当前最重要的是稳定完成这件事：

```text
上传项目 -> 判断能力 -> 发布链接 -> 运行起来 -> 用户试用 -> 收集反馈 -> 更新版本
```

如果一个架构设计不能让这条链路更稳定、更清楚、更容易维护，就不要做。

### 13. 早期应该更早做模块边界设计

如果重新来，我会在 MVP 阶段就做一个轻量但可扩展的模块边界，而不是把大量逻辑放在一个大 service 或一个大入口文件里。

更理想的结构应该是：

```text
API 层：只负责请求、权限、参数和响应
应用服务层：负责发布、更新、运行、诊断等业务编排
领域服务层：负责项目识别、内容安全、额度、数据库、运行器等单一能力
基础设施层：负责文件、Docker、MySQL、邮件、Nginx、外部 API
前端展示层：只呈现状态、操作和诊断，不承载复杂判断
```

这样做的价值不是一开始就做复杂系统，而是避免后面每加一个能力，都继续塞进原来的大 service 里。

但也要注意：不要为了拆分而大拆分。后续应该采用“跟着新能力逐步拆”的方式，例如 v1.0.0 如果做完整应用试用闭环，就可以顺手梳理发布编排、运行状态、数据库初始化和失败诊断的边界。

## 七、给下一个 AI 的复盘建议

接手后不要只问“下一个功能是什么”，要先问：

- 这个功能是否让用户更容易发布产品？
- 是否让试用者更容易打开和反馈？
- 是否减少了用户需要理解的技术概念？
- 是否让失败后的修复更清楚？
- 是否推进了完整应用试用部署闭环？

如果答案是否定的，这个功能就不应该优先做。

下一个阶段最应该避免的错误是：继续做一堆看起来有用、但没有形成能力跃迁的小功能。

DemoGo 接下来最需要的是一条可以被反复验证的主链路：

```text
完整应用样本 -> DemoGo 发布 -> 链接可访问 -> 数据可写入 -> 状态可诊断 -> AI 可修复 -> 版本可更新
```

这条链路跑通，DemoGo 才真正从 MVP 走向可商业试用。
