# DemoGo 产品需求文档（执行版 v3.0）

更新时间：2026-05-12  
适用阶段：v0.1.13 起  
文档性质：结合现有 v0.1.12 实际能力和两份 v2 文档后的可执行产品方案

---

## 1. 文档结论

DemoGo 当前已经不是“只有官网页面”的概念产品，而是一个已经可以注册、上传、检测、发布、管理 Demo 的可用 MVP。

但它距离“可以放心给真实用户试运营”的状态还有明显差距：

- 首页还不够打中用户痛点；
- 用户端还不像成熟产品工作台；
- 发布流程缺少过程感和结果页；
- 项目详情和部署记录不够清楚；
- 管理后台还偏数据列表，不像运营待办中心；
- 上传格式只支持 zip，覆盖面不够；
- 当前不支持完整后端、数据库、Node.js/Next.js 全栈应用，不能在文案中超前承诺。

因此，v0.1.13 的产品目标应调整为：

> 把 DemoGo 从“功能能用”打磨成“用户愿意用、运营人员方便管”的试运营版本。

---

## 2. 产品定位

### 2.1 当前定位

DemoGo 是一个面向 AI 生成项目的 Demo 发布与试用链接平台。

它帮助用户把 AI 工具生成的网页、活动页、H5 页面、前端原型和常见前端项目，快速变成一个可打开、可转发、可试用的链接。

### 2.2 不建议使用的定位

当前阶段不建议说：

- 支持任何 AI 产品；
- 支持完整全栈应用；
- 支持 Node.js 后端应用；
- 支持 Next.js SSR；
- 支持数据库应用；
- 支持 Docker 应用；
- 一键发布所有项目到互联网。

这些表达会造成用户预期过高，增加试运营风险。

### 2.3 推荐口号

主口号：

> AI 做出了产品雏形，别再卡在部署和打开这一步。

辅助表达：

> DemoGo 帮你绕开买服务器、配域名、研究部署平台这些麻烦，把 AI 生成的网页、活动页和产品原型快速变成可打开、可转发、可试用的 Demo 链接。

## 3. 目标用户与核心场景

| 用户类型 | 典型场景 | 核心需求 |
|---|---|---|
| 产品经理 | 用 AI 做了 MVP 页面，需要给领导、客户、同事试用 | 快速生成可打开链接 |
| 运营人员 | 活动页、报名页、预约页、留资页 | 快速上线、方便转发、后续需要数据回收 |
| 创业者 | 做了产品原型，需要验证需求 | 低成本试错、快速演示 |
| 设计师 | 作品集、展示页、交互原型 | 不想依赖开发部署 |
| 学生/个人开发者 | 课程作业、作品展示 | 简单、低成本、国内可访问 |

当前优先服务：

1. 产品经理；
2. 运营人员；
3. 独立创业者；
4. 使用 AI 编程工具的非技术用户。

## 4. 当前产品能力基线

v0.1.12 已具备：

- 首页；
- 注册登录；
- 用户端控制台；
- 管理后台；
- MySQL 主数据源；
- zip 上传；
- 项目检测；
- 静态页面发布；
- 常见前端源码项目自动构建；
- Demo 链接生成；

- 用户反馈；

- 发布记录；
- 复制转发文案；

暂不支持：

- HTTPS 正式域名；
- 完整日志流；
- 多租户资源隔离。

## 6. 首页需求

### 6.1 首页目标

首页不是功能说明书，而是转化页面。用户进入首页后应快速理解：

- DemoGo 是干什么的；
- 为什么我需要它；
- 它解决了我什么麻烦；
- 它和 Vercel、Netlify、自建服务器有什么区别；
- 我能不能马上试。

### 6.2 首页核心内容结构

建议结构：

1. 顶部导航；
2. 首屏痛点和 CTA；
3. 用户为什么卡住；
4. DemoGo 如何解决；
5. 三步发布流程；
6. 发布前基础体检；
7. 适用场景；
8. 支持与不支持边界；
9. 套餐；
10. 试用 CTA。

### 6.3 首屏建议文案

主标题：

```text
AI 做出了产品雏形，
别再卡在部署和打开这一步。
```

副标题：

```text
DemoGo 帮你绕开买服务器、配域名、研究部署平台这些麻烦，
把 AI 生成的网页、活动页和产品原型，
快速变成可打开、可转发、可试用的 Demo 链接。
```

CTA：

```text
立即生成试用链接
查看支持的项目类型
```

### 6.4 首页必须强调的差异点

| 用户麻烦 | DemoGo 表达 |
|---|---|
| 不会部署 | 上传项目包，生成试用链接 |
| 不想买服务器 | 试用阶段不用自己准备服务器 |
| 不会配域名 | 自动生成访问地址 |
| 国外平台国内访问不稳 | 面向国内试用链路 |
| 截图和录屏不能交互 | 生成可以真实点击的页面 |
| 表单/API 问题发布后才发现 | 发布前做基础体检和风险提示 |

### 6.5 首页不能写的内容

不能写：

- 支持任何产品；
- 支持 Node.js 全栈；
- 支持 Next.js SSR；
- 支持数据库；
- 支持支付系统；
- 已有大量真实客户评价；
- 3 秒上线；
- 100% 稳定。

没有真实用户评价前，不放虚构用户案例。可以先用“适用场景”替代。

## 7. 用户端需求

### 7.1 用户端整体结构

用户端应从“功能集合页”升级为“项目工作台”。

建议一级结构：

```text
工作台首页
创建项目
我的项目
项目详情
套餐与额度
升级申请
发布记录
反馈问题
账号设置
```

### 7.2 工作台首页

目标：让用户一进来就知道当前状态和下一步动作。

内容：

- 欢迎语；
- 当前套餐：Free / Lite / Pro；
- 使用情况：在线项目数、本月发布次数、Demo 有效期；
- 我的项目卡片；
- 创建新项目入口；
- 最近发布记录；
- 待处理升级申请提醒；
- 发布失败或风险提醒。

### 7.3 创建项目

创建项目应作为独立流程，不应只是一个上传区。

字段：

- 项目名称；
- 项目文件；
- 项目类型：自动检测为主，手动选择为辅；
- 高级设置：构建命令、输出目录，先作为可选项；
- 发布密码：内测期保留。

支持文件：

- `.zip`
- `.tar.gz`
- `.tgz`

文件限制：

- 最大 50MB；
- 文件数量限制；
- 解压后大小限制；
- 阻止敏感文件；
- 忽略无关目录。

### 7.4 项目类型

当前可对外承诺：

| 类型 | 当前支持情况 |
|---|---|
| 纯 HTML | 支持 |
| dist/build 构建产物 | 支持 |
| React/Vue/Vite 源码 | 支持，需能构建为静态文件 |
| 活动页/报名页前端 | 支持页面发布，但表单数据暂不托管 |
| Node.js 后端应用 | 暂不支持 |
| Next.js SSR | 暂不支持 |
| 数据库应用 | 暂不支持 |
| Docker 应用 | 暂不支持 |

### 7.5 发布过程页

发布过程要让用户看到系统正在做什么。

v0.1.13 可先做“结构化步骤展示”，不一定做真正实时日志。

步骤：

```text
接收文件
解压文件
检查安全风险
检测项目类型
执行构建（如需要）
复制发布文件
生成访问链接
发布完成
```

每一步应有：

- 成功；
- 失败；
- 跳过；
- 说明。

### 7.6 发布成功页

发布成功后必须有结果页。

内容：

- 发布成功提示；
- Demo 名称；
- 访问地址；
- 有效期；
- 项目类型；
- 复制链接；
- 复制转发文案；
- 立即访问；
- 查看项目详情；
- 下一步建议。

分享文案应包含：

- Demo 名称；
- Demo 链接；
- 试用说明；
- DemoGo 平台传播语。

文案不能夸大，例如不写“30 秒上线”，除非后续有真实统计。

### 7.7 项目详情页

项目详情是 DemoGo 未来产品结构的核心。

建议 Tab：

```text
概览
部署记录
检测报告
访问统计
设置
```

概览：

- 项目名称；
- 状态；
- 类型；
- 访问链接；
- 创建时间；
- 更新时间；
- 有效期；
- 当前版本；
- 文件大小；
- 访问次数。

部署记录：

- 发布时间；
- 操作类型：发布/更新；
- 状态；
- 版本；
- 错误原因；
- 查看检测报告。

检测报告：

- 入口文件；
- 构建配置；
- 表单字段；
- 本地 API；
- 风险提示；
- AI 编程工具修复提示词。

设置：

- 修改名称；
- 重新部署；
- 下线；
- 恢复上线；
- 删除。

## 8. 管理后台需求

### 8.1 管理后台定位

管理后台不是简单数据列表，而是运营人员处理问题的工作台。

核心目标：

- 看今天有什么事要处理；
- 找到高价值用户；
- 找到发布失败和风险 Demo；
- 处理升级申请；
- 处理用户反馈；
- 判断产品试运营情况。

### 8.2 运营首页

优先显示：

- 待处理升级申请；
- 待处理反馈；
- 今日新增用户；
- 今日发布次数；
- 发布失败数量；
- 高风险 Demo 数量；
- Free / Lite / Pro 用户分布。

### 8.3 升级申请管理

保留当前原则：

- 用户端提交申请；
- 管理后台集中处理；
- 开通或拒绝；
- 拒绝原因页面内输入；
- 用户端展示处理结果。

不恢复“直接改用户套餐”的入口，避免流程混乱。

### 8.4 用户管理

v0.1.13 先做只读强化：

- 搜索用户；
- 查看套餐；
- 查看 Demo 数；
- 查看注册时间；
- 查看最近操作；
- 查看升级申请和反馈。

后续再做用户详情页和禁用/启用。

### 8.5 Demo 管理

应支持：

- 搜索 Demo；
- 按状态筛选；
- 查看用户；
- 查看访问次数；
- 查看检测风险；
- 查看部署记录；
- 强制下线；
- 删除已下线 Demo。

### 8.6 反馈管理

应支持：

- 查看反馈列表；
- 按状态筛选；
- 更新状态；
- 关联用户；
- 关联 Demo；
- 标记处理结果。

## 9. 套餐方案

当前统一使用：

```text
Free
Lite
Pro
```

当前建议维持：

| 套餐 | 在线 Demo | 月发布/更新次数 | Demo 有效期 |
|---|---:|---:|---:|
| Free | 1 | 3 | 7 天 |
| Lite | 3 | 20 | 30 天 |
| Pro | 10 | 60 | 30 天 |

当前不建议马上引入：

- 个人版/团队版命名；
- 自动支付；
- 自动续费；
- 邀请续期；
- 无限额度。

原因：当前还在试运营验证阶段，应优先验证用户是否愿意为“更高额度和更长试用”付费。

## 10. 邀请续期功能判断

邀请续期有价值，但不建议放在 v0.1.13。

原因：

- 当前产品体验还不够好，不宜过早做裂变；
- 邀请系统会带来防作弊、邮箱验证、注册风控、奖励规则等复杂度；
- 真实用户价值还没充分验证；
- 当前更重要的是提高首次使用成功率。

建议阶段：

```text
v0.1.13：不做邀请续期
v0.1.14 或 v0.2：视真实试运营数据决定是否做
```

若后续做邀请系统，应先做简单版：

- 每个用户生成邀请链接；
- 被邀请人注册后记录来源；
- 先只统计，不自动发奖励；
- 观察是否有真实传播价值；
- 再做续期奖励。

## 12. 验收标准

v0.1.13 完成后，应满足：

1. 新用户看首页后能理解 DemoGo 解决什么问题；
2. 新用户登录后知道第一步该创建项目；
3. 上传 `.zip / .tar.gz / .tgz` 都能被识别；
4. 发布成功后用户能马上复制链接和转发文案；
5. 用户能进入项目详情查看发布记录和检测结果；
6. 管理员能从后台看到待处理事项；
7. 所有套餐名称统一为 Free / Lite / Pro；
8. 页面没有浏览器原生 `alert/prompt` 弹窗；
9. 不出现“支持 Node.js 全栈”这类超出当前能力的表述；
10. 发布前完成代码审查和自测。
