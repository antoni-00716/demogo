# DemoGo v0.1.13 版本计划：项目工作台与发布体验优化版

更新时间：2026-05-12  
版本定位：试运营体验打磨版  
前置版本：v0.1.12 已上线，React/Vite 前端迁移完成

---

## 1. 版本结论

v0.1.13 不建议继续零散修按钮和样式，而应围绕一个核心目标：

> 把 DemoGo 从“功能可用”打磨成“试运营用户愿意用”。

v0.1.12 已经完成前端技术迁移，但首页、用户端、管理后台仍然偏 MVP，用户路径不够顺，发布流程缺少过程感和结果页，运营后台也还不像真正的运营工作台。

v0.1.13 要解决的不是“有没有功能”，而是：

- 用户看首页后是否愿意注册；
- 注册后是否知道下一步该做什么；
- 上传发布是否有清晰流程；
- 发布成功后是否能马上转发；
- 用户是否能管理自己的项目；
- 运营人员是否能看到待处理事项；
- 系统支持边界是否说清楚。

---

## 2. 版本目标

### 2.1 产品目标

1. 首页更能打中用户痛点；
2. 用户端升级为项目工作台；
3. 创建项目流程独立清晰；
4. 发布过程有步骤反馈；
5. 发布成功页明确展示结果；
6. 项目详情页形成基础结构；
7. 管理后台升级为运营待办中心；
8. 上传格式支持 `.zip / .tar.gz / .tgz`；
9. 所有文案避免超出当前真实能力。

### 2.2 技术目标

1. 继续使用 React + Vite + TypeScript；
2. 后端继续保持 Express 单体，不拆微服务；
3. 短期继续用 `demos` 表承载项目概念；
4. 新增或强化部署事件结构，为发布过程和项目详情服务；
5. 抽象安全解压逻辑，支持 zip 和 tar.gz；
6. 不引入 Redis、PM2 用户应用托管、Next.js、K8s。

---

## 3. 不纳入本版本

v0.1.13 暂不做：

- 在线支付；
- 订单系统；
- 邀请续期；
- 邮箱验证码；
- 找回密码；
- 表单托管正式功能；
- 自定义域名；
- HTTPS 正式域名；
- Node.js/Next.js 后端应用托管；
- Docker 应用；
- Redis；
- 微服务拆分；
- K8s；
- 虚构用户案例。

## 4. P0 功能范围

### 4.1 首页继续打磨

目标：让真实用户一眼理解 DemoGo 的价值。

重点：

- 强化“AI 做出来了，但别人打不开/不能试用”的痛点；
- 强化国内访问稳定、无需买服务器、无需配域名、无需研究部署；
- 强调发布前基础体检；
- 明确支持边界；
- CTA 引导到创建项目；
- 不写 Node.js 全栈、Next.js SSR、数据库应用等暂不支持内容。

建议首页结构：

```text
首屏痛点
为什么需要 DemoGo
三步发布流程
发布前基础体检
适用项目类型
支持与暂不支持
套餐
最终 CTA
```

### 4.2 用户端工作台

目标：用户登录后不再面对一堆功能，而是看到自己的项目和下一步动作。

内容：

- 当前套餐；
- 在线项目数；
- 本月发布/更新次数；
- 最近项目卡片；
- 创建新项目入口；
- 最近发布记录；
- 待处理升级申请；
- 常见问题反馈入口。

### 4.3 创建项目流程

目标：把上传发布变成一个用户能理解的流程。

字段：

- 项目名称；
- 上传文件；
- 支持格式 `.zip / .tar.gz / .tgz`；
- 项目类型：默认自动检测；
- 高级设置：构建命令、输出目录，先可选展示；
- 发布密码：内测期保留。

交互：

- 支持拖拽上传；
- 文件选择后展示文件名、大小、格式；
- 点击“开始检测/部署”前展示支持边界；
- 错误提示用人话表达。

### 4.4 上传格式扩展

新增支持：

```text
.tar.gz
.tgz
```

安全要求：

- 禁止路径穿越；
- 禁止绝对路径；
- 禁止软链接和硬链接；
- 限制文件数量；
- 限制解压后总大小；
- 阻止敏感文件；
- 忽略无关目录；
- 异常压缩包返回明确错误。

### 4.5 发布过程展示

v0.1.13 可先做“结构化步骤”，不要求真正实时日志。

步骤：

```text
接收文件
解压文件
安全检查
检测项目
执行构建（如需要）
发布文件
生成访问地址
发布完成
```

每一步展示：

- 状态；
- 时间；
- 说明；
- 失败原因。

### 4.6 发布成功页

发布成功后，用户应看到一个明确结果页。

内容：

- 发布成功；
- Demo 名称；
- 访问地址；
- 有效期；
- 项目类型；
- 复制链接；
- 复制转发文案；
- 立即访问；
- 查看项目详情；
- 下一步建议。

### 4.7 项目详情页基础版

项目详情页建议包括：

```text
概览
发布记录
检测报告
访问统计
设置
```

v0.1.13 可先实现基础版：

- 项目信息；
- 访问链接；
- 状态；
- 访问次数；
- 版本；
- 最近发布记录；
- 最近检测报告；
- 下线/恢复/删除/更新。

### 4.8 管理后台运营待办

目标：运营人员打开后台后先看到“今天要处理什么”。

内容：

- 待处理升级申请；
- 待处理反馈；
- 最近发布失败；
- 高风险 Demo；
- 今日新增用户；
- 套餐分布；
- 最新 Demo。

## 6. 数据库与接口调整

### 6.1 短期原则

产品上可以逐步叫“项目”，但技术上 v0.1.13 不建议立刻把 `demos` 表改名为 `projects`。

原因：

- 当前线上数据已经存在；
- API 和前端已依赖 Demo 模型；
- 改名会带来较大迁移风险；
- 对用户体验提升不明显。

### 6.2 建议新增表

建议新增：

```text
deployment_events
```

用途：

- 记录发布过程步骤；
- 展示发布过程；
- 支撑项目详情发布记录；
- 记录失败原因；
- 后续升级为真正 deployment 模型。

### 6.3 建议新增接口

```text
GET /api/demos/:id
GET /api/demos/:id/events
GET /api/demos/:id/inspection
```

是否新增 `POST /api/projects` 暂缓。  
当前可继续使用 `/api/deploy` 创建 Demo。

## 8. 设计与体验原则

### 8.1 总体风格

继续使用：

- 白色干净 SaaS；
- 青绿色主色；
- 深蓝/深青结构色；
- 清晰表格和卡片；
- 克制、专业、可信。

避免：

- 过度渐变；
- AI 科技风紫黑界面；
- 过多装饰；
- 大面积营销空话；
- 卡片套卡片；
- 按钮文字太长；
- 浏览器原生 alert/prompt。

### 8.2 交互原则

- 用户每一步都知道自己在哪里；
- 上传前知道支持什么；
- 检测后知道能不能发布；
- 发布中知道系统在做什么；
- 发布后知道下一步该干什么；
- 失败后知道怎么修；
- 管理员知道哪些事情需要处理。

## 9. 风险与控制

| 风险 | 控制方式 |
---|---|
| tar.gz 解压安全风险 | 统一安全解压层，禁止路径穿越和软链接 |
| 页面改动范围较大 | 分模块实现，先保证核心发布链路 |
| 工作台复杂度上升 | 先做基础版，不一次性做完整路由系统 |
| 后端表结构变动 | 只新增，不大迁移 |
| 用户预期过高 | 首页和检测报告明确支持边界 |
| 发布流程中断 | 保留现有 `/api/deploy` 能力，不推倒重来 |

## 10. 验收标准

v0.1.13 验收通过标准：

1. 首页主路径清晰；
2. 用户登录后看到工作台；
3. 用户能创建项目并上传 `.zip / .tar.gz / .tgz`；
4. 上传后能看到检测/发布步骤；
5. 发布成功后出现成功页；
6. 用户能复制链接和转发文案；
7. 用户能查看项目详情；
8. 管理后台能看到运营待办；
9. 升级申请、反馈、Demo 管理不回退；
10. 后端健康检查正常；
11. 前端构建通过；
12. 后端 check 通过；
13. smoke test 通过；
14. 部署包和回滚脚本正常。
