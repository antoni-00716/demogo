# 当前状态

当前线上版本以 `https://demogo.cn/api/health` 返回为准。

当前本地开发版本以根目录 `VERSION` 为准。开发完成不等于已经上线。

## 产品定位

DemoGo 用来把开发出来的产品快速发布成可访问、可分享、可试用的链接。核心闭环是：

```text
项目文件或源码 -> DemoGo 检查 -> 生成试用链接 -> 分享给用户 -> 收集反馈
```

## 已具备能力

- 网页上传生成试用链接。
- AI 工具通过 CLI、MCP、Codex Skill 或 Agent API 发布。
- AI/CLI/MCP 可以更新已有项目，原链接保持不变。
- CLI 在同一项目目录会保存 `.demogo/project.json`，下次 `deploy` 默认更新原链接。
- Free 首次发布使用随机链接。
- Lite/Pro 可在用户端修改链接后缀。
- Pro 可提交 `xxx.demogo.cn` 二级域名申请。
- 支持 Node.js 单服务试用运行环境，前提是线上运行器已开启。
- 支持为符合条件的 Node.js 项目分配 MySQL 空试用数据库，前提是线上数据库能力已开启。
- 支持项目识别 2.0：能识别常见前端框架、Node 后端框架、数据库/ORM、环境变量线索，并告诉用户当前能否发布、缺什么、下一步让 AI 怎么改。
- 本地 v0.5.2 已支持结构化失败诊断：失败后给出类别、证据、处理动作和给 AI 的修复提示。
- 本地 v0.5.3 已支持 Supabase 外部后端连接闭环：识别、配置、基础连接检测、环境变量注入和管理端展示。
- 本地 v0.6.0 已新增复杂应用试用闭环验收：综合判断页面、后端、运行环境、数据库/外部后端、配置、版本更新和失败诊断。
- 本地 v0.6.1 已新增复杂应用样本验收包：用“轻电商/商品预约”样本持续验收前端、Node.js、MySQL、schema 初始化、接口和版本更新链路。
- 本地 v0.6.2 已新增复杂应用失败样本库：覆盖缺 start、缺运行配置、未监听端口、依赖安装失败、Redis 不支持等失败路径，并强化诊断分类。
- 本地 v0.7.0 已新增试用交付报告：直接判断项目是否可以分享给用户试用、是否可以开始收集反馈，并补强完整应用验收矩阵和 schema.sql 失败诊断。
- 本地 v0.8.0 已开始 AI 编程工具插件化集成：用户端 AI 发布页统一为一条通用提示词，并新增 Codex Plugin 雏形。

- 管理端可查看用户、项目、发布失败原因、内容检查、运行环境、数据库和失败诊断等运营信息。

## 明确边界

DemoGo 当前不是完整云平台，不承诺生产级长期托管。

当前不支持：

- 多服务应用。
- Redis、MongoDB、PostgreSQL 自动托管。
- Supabase 只支持连接用户自己的项目，不托管、不自动创建、不自动执行外部数据库迁移。
- Python/Java/Go 后端。
- 仅支持符合单服务条件的 Next.js、Nuxt、TanStack Start 运行；Remix、SvelteKit、Astro 等完整应用运行态暂不支持。
- WebSocket、真实支付、生产级登录系统托管。
- 自动 DNS/SSL 自助绑定企业独立域名。

## 版本规则

- 开发完成不等于发布。
- 只有项目负责人明确说“发布这个版本”时，才进行线上部署和 npm 发布。
- 每次涉及 CLI 版本更新并实际发布线上版本时，必须同步验证 npm 和 npx。
