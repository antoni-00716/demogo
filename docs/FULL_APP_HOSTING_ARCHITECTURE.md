# DemoGo 完整应用部署能力架构方案

## 目标

DemoGo 的目标不是做电商平台，而是具备部署完整应用的能力。

完整电商平台项目只是验收边界，因为这类项目通常同时包含：

- 前端页面。
- 后端 API。
- 数据库。
- 管理后台。
- 环境变量。
- 初始化脚本。
- 运行日志和错误诊断。

如果 DemoGo 能让这类项目快速生成可试用链接，就说明平台的部署能力已经从“网页发布工具”升级为“完整应用试用部署平台”。

## 产品边界

### 当前应支持

- 静态网页和单 HTML 页面。
- 前端构建产物，例如 `dist`、`build`、`out`、`public`。
- 可构建成静态页面的前端源码项目。
- Node.js 单服务试用环境。
- Node.js + MySQL 空试用数据库。
- AI/CLI/MCP/API 发布和更新已有链接。

### 下一阶段重点支持

- 前端 + 后端的一体化项目。
- 后端 + MySQL 的试用数据库项目。
- 前端 + 用户自有 Supabase 的项目。
- 带数据库初始化脚本的项目。
- 带必要环境变量的项目。
- 接近完整应用结构的项目。

### 暂不支持

- 多服务编排。
- Docker Compose。
- Redis、MongoDB、PostgreSQL 自动托管。
- Python、Java、Go 后端。
- Remix/SvelteKit/Astro 等完整应用运行态。
- WebSocket 长连接。
- 真实支付、生产级登录系统和生产数据库托管。
- 高可用、弹性扩容、长期生产运行。

这些不是永久不做，而是在 Node.js + MySQL 主链路稳定前不应分散资源。

## 项目识别模型

用户上传项目后，DemoGo 应自动识别项目类型，尽量不让用户先理解技术细节。

识别顺序建议如下：

1. 是否已有静态入口。
   - 根目录或构建目录存在 `index.html`。
   - 可直接发布为静态试用链接。

2. 是否是前端源码项目。
   - 存在 `package.json`。
   - 存在 `vite`、`react-scripts`、`vue-cli-service` 等构建依赖或构建命令。
   - 能执行构建并生成静态产物。

3. 是否是 Node.js 后端项目。
   - 存在 `package.json`。
   - 存在 `start` 或 `start:prod` 命令。
   - 存在 `server.js`、`app.js`、`index.js`、`src/server.*` 等入口。

4. 是否需要数据库。
   - 依赖中出现 `mysql2`、`sequelize`、`prisma`、`typeorm` 等。
   - 依赖中出现 `@supabase/supabase-js` 或 Supabase 环境变量。
   - 项目中出现数据库连接配置。
   - 存在 `schema.sql`、`migrations/`、`prisma/schema.prisma`、`supabase/migrations/` 等初始化线索。

5. 是否缺少必要配置。
   - 需要环境变量但没有示例文件。
   - 有 `.env.example`、`.env.template` 时，引导用户填写。
   - 缺少启动命令、构建命令或数据库初始化入口时，明确提示失败原因。

## 完整应用部署流程

完整应用不应只返回一个链接，而应返回一组状态，让用户知道系统做了什么。

建议流程：

```text
上传项目
  -> 解压和安全过滤
  -> 项目类型识别
  -> 内容安全检查
  -> 构建前端或确认静态产物
  -> 准备后端运行环境
  -> 创建试用数据库
  -> 执行数据库初始化
  -> 注入运行环境变量
  -> 启动后端服务
  -> 生成访问链接
  -> 返回运行状态和失败诊断
```

不是所有项目都会走完整流程。静态网页只需要前几步，Node + MySQL 项目才需要后端和数据库步骤。

## 数据库能力设计

### 业务含义

数据库能力的目标是让用户的完整应用能被打开、操作和试用，不是承诺生产级数据库服务。

对外应表达为“试用数据库”或“演示数据空间”，避免让用户误解为正式生产数据库。

数据库能力分两类：

- DemoGo 内置试用数据库：平台为项目分配 MySQL 空库，用于短期试用。
- 外部后端连接：用户提供自己的 Supabase/Postgres 等服务，DemoGo 负责识别、配置和注入连接参数。

Supabase 是外部后端连接的优先对象。第一阶段只连接用户自己的 Supabase，不自动创建 Supabase 项目，不托管 Supabase，也不自动执行生产迁移。

### 技术逻辑

- 每个 Demo 分配独立数据库或独立数据空间。
- 数据库名称、账号、密码由平台生成。
- 连接信息只注入运行环境，不在用户界面直接展示完整密码。
- 删除 Demo 时同步删除或标记清理数据库。
- 支持重置数据库，用于恢复演示初始状态。

### 初始化方式

优先级建议：

1. 明确的 `schema.sql`。
2. `migrations/` 目录。
3. Prisma schema 和迁移命令。
4. ORM 初始化脚本。
5. 无初始化脚本时，只创建空数据库并提示用户。

MVP 阶段优先支持 `schema.sql` 和简单迁移提示，不要一开始覆盖所有 ORM。

Supabase 项目如检测到 `supabase/migrations/`，先提示用户在 Supabase 控制台或 Supabase CLI 中执行迁移。DemoGo 后续可评估是否提供自动迁移，但不能在未确认安全边界前直接写入用户外部数据库。

## 环境变量设计

完整应用经常需要环境变量。DemoGo 应把这件事变成用户能理解的配置步骤。

建议分为三类：

- 平台自动生成：`PORT`、`DATABASE_URL`、数据库账号密码。
- 用户必须填写：第三方 API Key、短信、邮件、支付、AI 模型密钥。
- 外部后端配置：Supabase URL、Supabase anon key、外部数据库连接参数。
- 暂不支持：生产支付回调、复杂对象存储、私有网络连接。

用户端展示方式：

- 告诉用户“这个项目需要补充配置才能运行”。
- 不直接暴露复杂技术细节。
- 给 AI 工具明确提示：需要补齐哪些变量、在哪里配置。
- 对 Supabase 项目明确提示：只能填写可公开用于前端的 anon key，不要填写 `service_role` 高权限密钥。

Supabase 变量名应按项目类型适配：

- Vite：`VITE_SUPABASE_URL`、`VITE_SUPABASE_ANON_KEY`。
- Next.js：`NEXT_PUBLIC_SUPABASE_URL`、`NEXT_PUBLIC_SUPABASE_ANON_KEY`。
- Node.js 服务端：`SUPABASE_URL`、`SUPABASE_ANON_KEY`。

## 用户端体验

用户上传后，应看到清晰的部署判断，而不是技术堆叠。

建议状态文案：

- “已识别为网页项目，可以直接生成试用链接。”
- “已识别为前端项目，DemoGo 将先生成网页版本。”
- “已识别为后端应用，DemoGo 将准备独立试用环境。”
- “已识别为完整应用，需要后端和试用数据库。”
- “项目缺少启动命令，请让 AI 为项目补充 start 命令。”
- “项目需要数据库初始化脚本，请让 AI 补充 schema.sql 或迁移说明。”

发布成功后，项目详情应展示：

- 访问链接。
- 项目类型。
- 前端状态。
- 后端运行状态。
- 数据库状态。
- 最近一次发布版本。
- 更新版本入口。
- 日志和失败诊断入口。

## 管理端体验

管理端需要支撑运营判断和成本控制。

建议展示：

- 每个 Demo 的项目类型。
- 是否使用运行容器。
- 是否使用数据库。
- 容器状态和最近启动时间。
- 数据库状态和创建时间。
- 发布失败原因分布。
- 资源占用概览。
- 高风险项目和异常运行项目。

后续做套餐时，可以基于这些数据设计权益：

- Free：静态和轻量试用为主。
- Lite：更多发布次数、可改链接后缀、更多运行时长。
- Pro：后端运行、试用数据库、二级域名申请、更高资源额度。

具体套餐权益应在商业化方案中单独确认，不在本方案里一次定死。

## 技术架构建议

后端应从单一发布逻辑逐步变成“部署编排器”。

核心模块建议：

- 项目识别器：判断项目类型和所需能力。
- 构建器：处理前端源码构建。
- 运行器：管理 Node.js 容器生命周期。
- 数据库管理器：创建、初始化、重置、删除试用数据库。
- 环境变量管理器：生成、保存、注入项目运行配置。
- 日志诊断器：收集启动失败、构建失败、数据库失败原因。
- 状态同步器：服务重启后恢复容器和数据库状态认知。

当前已有模块可以继续演进，不建议推倒重来。

结合主流 AI 编程工具调研，完整应用架构不能只围绕 MySQL 设计。更合理的主链路是：

```text
静态/前端构建
  -> Node 单服务
  -> Node SSR/元框架运行态
  -> 环境变量和密钥配置
  -> MySQL 试用数据库
  -> Supabase/Postgres 外部后端连接
  -> 完整应用样本验收
```

其中 Supabase/Postgres 不一定一开始由 DemoGo 自建，但必须先在产品和技术架构中承认它们是 AI App Builder 高频产物的一部分。

这意味着 DemoGo 后续不是只有一条 `Node + MySQL` 路线，还要有一条 `前端 + Supabase` 路线。前者由 DemoGo 提供试用环境，后者由 DemoGo 帮用户把外部后端接起来。

## 失败诊断原则

完整应用部署最重要的不是一次成功，而是失败后能让用户知道下一步怎么改。

失败提示必须回答：

- 哪一步失败。
- 为什么失败。
- 对用户有什么影响。
- 让 AI 工具怎么改。

典型失败包括：

- 没有静态入口。
- 没有构建命令。
- 没有启动命令。
- 端口没有使用平台提供的 `PORT`。
- 缺少数据库初始化脚本。
- 缺少必要环境变量。
- 启动超时。
- 内容安全不通过。
- 资源额度不足。

## 后续版本建议

### v0.5.0 项目识别 2.0

- 强化项目识别器。
- 识别 Vite、Next.js、TanStack Start、Nuxt、SvelteKit、Astro。
- 识别 Express、Fastify、Hono。
- 识别 Supabase、Postgres、MySQL、Prisma、Drizzle。
- 用户端展示识别结果、当前是否可发布和缺失条件。

### v0.5.1 完整应用运行闭环

- 支持符合条件的 Node.js 单服务和部分 SSR/元框架项目进入运行环境。
- 支持运行配置保存、脱敏展示和缺失配置后补齐。
- 支持 MySQL 试用数据库 `schema.sql` 初始化和重置。
- 用户端和管理端展示运行配置、数据库初始化状态和运行日志摘要。

### v0.5.2 失败诊断 2.0

- 建立统一失败诊断对象。
- 覆盖发布、更新、异步任务、运行重启、数据库初始化等失败场景。
- 区分额度、内容安全、项目包、不支持能力、运行配置、依赖安装、构建、启动、数据库初始化等类别。
- 用户端和管理端展示证据、处理动作和给 AI 的修复提示。

### v0.5.3 Supabase 外部后端连接闭环

- 将 Supabase 配置、检测、注入和诊断合并。
- 支持 Supabase 项目识别、用户端连接配置、基础连通性检测和外部后端诊断。
- 根据 Vite、Next.js、Node.js 等项目类型注入对应 Supabase 环境变量。
- 拒绝保存 `service_role` 高权限密钥。
- 管理端区分 DemoGo MySQL 试用库和用户自有 Supabase。
- 保持边界：不托管 Supabase，不自动创建 Supabase 项目，不自动执行外部数据库迁移。

### v0.6.0 完整应用样本验收

- 使用一个完整应用样本验证能力。
- 样本可以接近电商平台结构，但 DemoGo 不开发电商业务。
- 验收前端、后端、数据库、管理后台、API 路由、更新版本是否可用。
- 至少验收两类样本：`前端 + Node.js + MySQL`，以及 `前端 + Supabase`。

## 关键取舍

- 先把 Node.js + MySQL 主链路做稳，同时补上 Supabase 外部后端连接这条高频链路。
- 先做试用部署，不承诺生产级托管。
- 先做单服务完整应用，不做多服务编排。
- 先让失败可理解，再追求更高自动化。
- 先服务真实试用和反馈，不急于做复杂云平台功能。
