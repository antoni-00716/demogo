# DemoGo v0.2.0 开发方案：AI 工具一键发布正式体验版

日期：2026-05-18

## 版本定位

v0.2.0 是 DemoGo 从“网页上传发布工具”走向“AI 编程工具发布平台”的关键版本。

核心目标：

让用户在 Codex、Cursor、Claude Code、Hammers Agent、OpenCode 等 AI 编程工具里说“发布到 DemoGo”，AI 就能完成项目检查、打包、上传、内容检查、生成试用链接，并把链接返回给用户。

## 本版主线

本版只围绕一条主线：

```text
AI 编程工具
  -> 读取 DemoGo 发布说明
  -> 使用 DemoGo AI 发布口令
  -> 打包当前项目
  -> 调用 DemoGo 发布接口
  -> 返回试用链接或清楚的失败原因
```

## 核心任务一：AI 发布链路正式化

### 要做

- CLI / MCP / Codex Skill 统一升级到 `0.2.0`。
- CLI 默认 API 地址不再固定服务器 IP。
- CLI 配置逻辑改为：
  - 优先命令参数。
  - 其次本地配置。
  - 其次环境变量。
  - 都没有时提示用户填写 DemoGo 平台地址。
- 明确支持命令：
  - `demogo config set --api <DemoGo地址> --token <AI发布口令>`
  - `demogo config show`
  - `demogo deploy`
- 不新增不存在的 `demogo inspect` 命令，避免误导。
- CLI 发布失败时清楚说明：
  - 项目结构问题。
  - 内容检查问题。
  - 套餐额度不足。
  - 后端能力暂不支持。
  - 服务端处理异常。
- MCP 工具继续复用 Agent 发布 API，不重写发布逻辑。
- Codex Skill 明确告诉 AI：
  - 什么项目能发。
  - 什么项目不能发。
  - 如何配置 API 地址和发布口令。
  - 发布失败后如何修改项目。

### 不做

- 不做新的后端托管能力。
- 不做数据库托管。
- 不做复杂插件市场。

## 核心任务二：用户端 AI 发布页重做

### 要做

用户端增加或重做“让 AI 帮我发布”页面，用非技术语言解释：

- DemoGo AI 发布口令是什么。
- 为什么需要发布口令。
- 如何把发布口令交给 AI 工具。
- 如何让 AI 在项目目录里执行发布。
- 发布成功后用户会得到什么。
- 发布失败时 AI 应该怎么改。

页面提供三类可复制指令：

1. 给 Codex 的发布指令。
2. 给 Cursor / Claude Code 的发布指令。
3. 给其他 AI Agent 的通用发布指令。

页面区分三种发布方式：

- 网页上传：适合非技术用户手动上传。
- CLI 发布：适合让 AI 在项目目录里执行。
- MCP / Skill 发布：适合深度集成。

### 用户可见表达原则

不要说：

- API 调用链。
- token 哈希。
- multipart。
- Agent API。
- MCP 协议细节。

应该说：

- “把这段指令发给你的 AI 编程工具。”
- “AI 会帮你打包并生成 DemoGo 试用链接。”
- “发布口令只用于生成链接，不要发到公开页面。”
- “如果发布失败，AI 会根据提示修改项目后再试。”

## 核心任务三：真实试用准备包

### 要做

准备真实试用所需材料：

- 5-10 个测试项目样例清单。
- 用户试用说明。
- AI 工具发布说明。
- 后台试用观察指标。

后台增加基础试用观察指标：

- 发布成功数。
- 发布失败数。
- 常见失败原因。
- 内容拦截数。
- 表单收集数。
- AI 发布次数。

这些指标不要求做复杂分析系统，先做到运营人员能快速判断试用是否顺畅。

## 后端改动范围

### 要做

- Agent 发布 API 保持现有路径：`POST /api/agent/deploy`。
- 增加 AI 发布来源统计字段或复用现有审计记录。
- 后台概览增加 AI 发布相关指标。
- 失败响应继续返回结构化信息，方便 AI 工具转述。

### 不做

- 不改发布核心架构。
- 不新增独立 AI 发布服务。
- 不绕过 v0.1.25/v0.1.26 的内容检查。

## CLI 改动范围

### 要做

- 版本升级到 `0.2.0`。
- 默认 API 地址从固定 IP 改为“必须配置或从环境读取”。
- 配置提示更适合非技术用户。
- 发布失败时保留服务端返回的修改建议。
- `help` 文案改为更清楚的真实使用路径。

### 验收

- `demogo --version` 返回 `0.2.0`。
- 未配置 API 地址时，提示用户配置 DemoGo 平台地址。
- 未配置发布口令时，提示用户到工作台生成 AI 发布口令。
- 配置完成后可以打包并调用 `/api/agent/deploy`。

## MCP 改动范围

### 要做

- 版本升级到 `0.2.0`。
- `demogo_get_config` 返回更清楚的配置状态。
- `demogo_check_project` 说明当前项目是否适合发布。
- `demogo_deploy_project` 继续复用 Agent 发布 API。
- 错误信息方便 AI 工具直接转述给用户。

### 不做

- 不引入复杂 MCP 鉴权。
- 不做多用户账号绑定。

## Codex Skill 改动范围

### 要做

- Skill 文案升级为正式体验说明。
- 明确 DemoGo 当前支持和不支持的项目。
- 明确内容检查不能绕过。
- 明确失败后应该修改项目，而不是反复重试。
- 明确 CLI 优先、MCP/API 兜底。

## 用户端改动范围

### 要做

- AI 发布页重做。
- 发布口令生成、重置、隐藏明文逻辑保持不变。
- 增加可复制 AI 指令。
- 说明 CLI / MCP / Skill 三种方式的区别。

### 不做

- 不大改首页视觉。
- 不大改项目列表。
- 不重做套餐页面。

## 管理后台改动范围

### 要做

- 概览增加 AI 发布次数。
- 概览增加发布成功/失败粗略指标。
- 失败原因先基于现有发布事件和审计记录归类。

### 不做

- 不做复杂数据分析。
- 不做图表大屏。

至少覆盖：

- 前端生产构建。
- CLI 版本检查。
- CLI 配置检查。
- CLI 未配置 API 地址提示。
- CLI 未配置发布口令提示。
- CLI 打包逻辑。
- MCP initialize。
- MCP tools/list。

- v0.2.0 打包产物检查。

## 验收标准

- 用户能在工作台生成 AI 发布口令。
- 用户能复制一段指令给 Codex / Cursor / Claude Code。
- AI 工具能理解如何发布到 DemoGo。
- CLI/MCP/Skill 版本统一为 `0.2.0`。
- 正常项目能生成试用链接。
- 内容风险项目会被拦截。
- 发布失败时 AI 能看到原因和修改建议。
- 后台能看到 AI 发布相关基础指标。
- 生成 v0.2.0 部署包。

## 不做事项

- 不做后端项目托管。
- 不做数据库托管。
- 不做支付、订单、登录系统托管。
- 不做团队协作。
- 不做大规模三端视觉重构。

## 风险与取舍

1. AI 工具差异较大，不能保证所有工具都像 Netlify 插件一样自动调用。
   - 处理方式：先把 CLI/MCP/Skill 三条路径做好，让主流 AI 工具有明确可执行方式。

2. 默认 API 地址去 IP 后，用户需要配置平台地址。
   - 处理方式：用户端复制指令中自动带上当前平台地址。

3. CLI/MCP/Skill 升级到 `0.2.0` 会扩大测试范围。
   - 处理方式：本版重点测试 AI 发布链路，不夹杂其他大功能。

## 建议开发顺序

1. 更新 CLI/MCP/Skill 版本和配置逻辑。
2. 重做用户端 AI 发布页。
3. 增加后台 AI 发布观察指标。
4. 更新打包脚本和交付包。
5. 补充测试和报告。
