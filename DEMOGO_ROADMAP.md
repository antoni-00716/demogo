# DemoGo 路线图与下一版本范围

更新时间：2026-05-11

## 1. 路线图原则

DemoGo 的路线要兼顾两件事：

1. 架构上为后续商业化、AI 能力、表单托管和微服务拆分留出口；
2. 实施上保持低成本、小步快跑、每个版本都能上线验证。

不要在 v0.1.x 阶段直接追求完整微服务、完整支付、完整后端应用发布。

建议总路线：

> 可用 MVP → 准商业化试用 MVP → 数据回收 MVP → 商业化验证 → 可规模化架构。

从 v0.1.10 开始，路线图增加一条长期原则：

> 每个版本都要同时推进“产品能力”和“架构能力”。

具体要求：

- 产品能力解决真实用户、运营、商业验证中的明确问题；
- 架构能力参考 Vercel、Coolify、Dokploy、CapRover 等国际成熟平台的通用做法；
- 不盲目追求微服务，不一次性重构；
- 通过小版本逐步迁移数据库表结构、代码分层、测试体系、部署体系；
- 每个版本至少偿还一块架构债。

## 2. 当前基线：v0.1.6

线上版本：`v0.1.6`

已具备：

- 用户注册登录；
- zip 上传；
- 项目检测；
- 静态页面和常见前端项目发布；
- Demo 链接生成；
- Demo 更新、下线、恢复、删除；
- 基础访问次数和估算流量；
- 管理后台查看和管理 Demo；
- Free/Lite/Pro 套餐规则在代码中预置；
- 阿里云 ECS + Nginx + Node.js Express 部署。

主要不足：

- 使用 JSON 作为主数据源；
- 没有正式数据库表结构；
- 发布体验和支持边界仍需优化；
- AI 能力较弱；
- 表单数据托管未实现；
- 管理后台不能调整套餐；
- 真实访问流量统计未完成；
- 域名和 HTTPS 尚未完全形成正式入口。

## 3. v0.1.7：准商业化地基版

### 3.1 版本目标

让 DemoGo 从“能跑的 MVP”升级为“可以小范围真实试用的准商业化地基版”。

核心目标：

- MySQL 数据库地基；
- 发布体验更清楚；
- 规则体检报告；
- 为 v0.2 表单托管预留接口和表结构。

### 3.2 建议范围

#### P0：数据库与数据模型

- 引入数据库作为主数据源；
- 建立第一批核心表；
- 从 JSON 迁移现有用户、会话、Demo、日志；
- 保留 JSON 备份；
- 新增数据库备份和回滚说明。

优先表：

- `users`
- `sessions`
- `plans`
- `demos`
- `demo_versions`
- `project_inspections`
- `audit_logs`

#### P0：发布体验优化

- “开始发布”明确说明会自动先检测；
- 检测失败展示结构化原因；
- 明确支持静态页面、前端项目；
- 明确暂不支持完整后端、数据库、Docker；
- 发布成功后生成更清楚的结果说明。

#### P0：项目检测增强

- 展示入口文件；
- 展示是否有 `package.json`；
- 展示是否有 build 命令；
- 展示忽略文件和阻止文件；
- 检测疑似表单字段；
- 检测疑似本地 API 调用。

#### P1：规则体检报告基础版

不接大模型，先由规则生成体检报告。

输出内容：

- 项目类型；
- 主要功能；
- 发布可行性；
- 表单/API 风险；
- 修复建议；
- 给 AI 编程工具的提示词。

大模型总结和更完整的 AI 体检报告后置到 v0.1.8 或后续版本。

### 3.3 不纳入 v0.1.7

- 完整在线支付；
- 管理后台套餐调整；
- 大模型 AI 体检；
- Docker 应用发布；
- 完整后端应用运行；
- 飞书/腾讯文档集成；
- 复杂多角色后台；
- 全量微服务拆分；
- 大规模前端重构。

## 4. v0.2.0：表单托管与数据回收版

### 4.1 版本目标

让 DemoGo 支持“页面可打开，数据能回来”。

### 4.2 核心功能

- 用户创建数据表/表单；
- DemoGo 生成标准提交 API；
- 支持报名、预约、留言、反馈、订单意向数据；
- 表单字段配置；
- 表单提交数据保存；
- 用户端查看、搜索、删除、导出 CSV；
- AI 生成接入 DemoGo 表单托管的修改指令。

### 4.3 数据表

- `forms`
- `form_fields`
- `form_submissions`

### 4.4 关键边界

支持标准表单数据托管。

不支持：

- 用户自定义后端 API 自动托管；
- 真实支付；
- 多表关联数据库；
- 复杂后台业务流程。

## 5. v0.2.x：运营增强版

目标：让 DemoGo 更适合真实运营。

建议内容：

- 真实访问统计；
- Nginx 日志解析或访问事件入库；
- 日统计汇总；
- 数据库自动备份；
- 表单提交限流；
- 管理员冻结 Demo 或表单；
- 域名和 HTTPS；
- 发布失败原因统计；
- 邮件通知或站内提醒。

## 5.1 v0.1.10：试用转化闭环修复版

目标：

把 DemoGo 从“能发布链接”推进到“用户能申请升级，你能处理申请，链接能更自然传播”。

产品功能：

- 用户端申请升级 Lite / Pro；
- 管理后台查看、筛选、开通、拒绝升级申请；
- 用户端展示最近升级申请状态；
- 额度耗尽时提示申请升级；
- 复制转发文案改为页面内面板，不再使用浏览器弹窗；
- 后台顶部显示待处理升级申请数；
- 管理员内部备注或申请处理备注。

架构演进：

- 新增 `plan_upgrade_requests` 正式表；
- 新增 `plan-request-service.js`；
- 升级申请进入 smoke test；
- 继续减少 `server.js` 中的业务职责；
- 管理后台运营待办逐步形成独立业务域。

不纳入：

- 在线支付；
- 订单系统；
- 表单托管正式版；
- 大模型 AI；
- Docker / 后端托管；
- 完整微服务拆分。

## 5.2 v0.1.12：前端现代化重构 + 试运营体验重设计版

目标：

把 DemoGo 从“功能能跑”推进到“可以拿给真实用户试运营”。本版本重点不是继续零散补功能，而是对首页、用户端和管理后台做一次系统性体验重设计，并启动 React + Vite + TypeScript 前端现代化迁移。

产品功能：

- 套餐名称统一为 Free / Lite / Pro；
- 用户端新增“我的升级申请”列表；
- 用户端展示待处理、已开通、已拒绝申请和管理员说明；
- 管理后台升级申请列表展示处理说明；
- 管理后台运营概览突出待处理申请和反馈；
- 用户端发布记录继续列表化；
- 首页重做为更适合真实试用转化的产品首页；
- 复制转发文案保持一键复制，不回退到复杂文本框体验。

页面改造：

- 首页：价值主张、三步流程、适用场景、支持边界、套餐、常见问题；
- 用户端：概览、我的 Demo、上传发布、套餐与额度、我的升级申请、本月发布记录、反馈；
- 管理后台：运营概览、待办事项、Demo 管理、用户列表、升级申请、反馈列表、最近事件、系统状态。

架构演进：

- 新建 `web/` 前端工程；
- 使用 React + Vite + TypeScript；
- 构建产物仍部署为静态文件；
- 保持 `/`、`/login.html`、`/app.html`、`/admin.html` 访问路径兼容；
- 统一套餐展示名和状态文案；
- 升级申请展示逻辑继续收敛到 service；
- 前端提取套餐配置对象；
- smoke test 增加“拒绝申请后用户端可见拒绝原因”；
- 继续为后续组件化和路由拆分做准备。

不纳入：

- 在线支付；
- 正式订单系统；
- 表单托管正式版；
- 大模型 AI；
- Docker / 后端托管；
- 完整微服务拆分。

## 6. v0.3.0：任务与 Worker 化

目标：把耗时和资源型任务从主 API 中拆出来。

建议内容：

- 引入任务队列；
- 构建 Worker；
- AI 体检 Worker；
- 统计汇总 Worker；
- 发布任务状态展示；
- 构建日志独立保存；
- 构建资源限制和隔离增强。

此阶段仍可以是模块化单体 + Worker，不必完整微服务。

## 7. v0.5+：规模化和商业化平台

触发条件：

- 真实付费用户出现；
- Demo 数量明显增长；
- 构建任务变多；
- 表单提交量变大；
- 运维复杂度上升。

建议内容：

- 对象存储 OSS；
- CDN；
- 支付订单；
- 发票和退款预留；
- 容器化构建环境；
- 按压力拆分微服务；
- 日志和监控平台；
- 灰度发布；
- OpenAPI、CLI、MCP、Codex Skill。

## 8. 微服务拆分路线

不要一开始做微服务，但要按可拆边界开发。

推荐拆分顺序：

1. Build Worker / Build Service；
2. AI Inspection Worker / AI Service；
3. Analytics Worker / Analytics Service；
4. Form/Data Service；
5. Billing Service；
6. Auth Service。

拆分依据不是“架构好看”，而是：

- 资源压力；
- 故障隔离；
- 团队协作；
- 部署频率；
- 数据边界。

## 9. 近期工作建议

下一步建议按这个顺序推进：

1. 确认数据库类型，优先检查服务器是否已有 MySQL；
2. 基于 `DEMOGO_DATA_MODEL.md` 定义 v0.1.7 实际落地表；
3. 设计 JSON 到数据库迁移方案；
4. 制定 v0.1.7 PRD 和开发任务；
5. 再进入代码开发。
