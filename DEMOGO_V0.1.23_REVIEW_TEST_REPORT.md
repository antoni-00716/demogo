# DemoGo v0.1.23 五轮代码审查与五轮测试记录

日期：2026-05-17

## 结论

v0.1.23 已完成 5 轮代码审查和 5 轮测试验证。自动化检查、AI 发布链路、CLI、MCP、部署包和主要边界场景均通过。

当前没有发现必须阻塞 v0.1.23 部署的问题。但有若干体验、工程和后续发布风险，建议放入下一版本优先修复。

## 五轮代码审查记录

### 第一轮：后端发布与数据链路

审查范围：

- 项目上传、检测、发布、更新、下线、恢复、删除。
- zip、tar.gz、tgz 解包。
- 源码构建、静态目录识别、SPA 路由回退。
- 自动表单识别、表单托管、公开提交。
- 套餐额度、升级申请、反馈、用户数据。

结果：

- 主链路完整，v0.1.23 的发布逻辑复用了现有检测、安全检查、额度检查和部署记录。
- `.zip`、`.tar.gz`、`.tgz` 的基础处理逻辑存在。
- 源码项目缺少 build 命令时可以前置拦截。
- 后端服务、SSR、数据库等不支持能力没有被误包装成已支持。

发现问题：

1. 后端会对同一个上传包做两次分析。
   - 位置：`server/src/server.js`
   - 说明：`handleCreateDeployment` 先执行 `inspectProjectArchive`，后续 `extractStaticDemo` 又会重新分析同一个包。
   - 影响：大包发布会更慢，用户等待时间变长。
   - 是否阻塞部署：否。

2. Free 用户下线项目后，自动表单可能仍占用表单额度。
   - 位置：后端额度与表单托管逻辑。
   - 说明：用户可能以为“下线项目”释放了相关能力，但表单能力仍可能影响后续开通。
   - 影响：非技术用户会困惑，不知道为什么新项目不能自动开表单收集。
   - 是否阻塞部署：否，属于体验问题。

### 第二轮：CLI 与 MCP

审查范围：

- `demogo` CLI 的配置、检查、发布、错误提示。
- MCP Server 的工具列表、项目检查、项目发布、配置读取。
- AI 发布密钥使用方式。
- CLI/MCP 与 Agent 发布 API 的复用关系。

结果：

- CLI 能独立执行 `config`、`deploy`、`--version`。
- MCP 能提供项目检查和发布工具。
- AI 发布链路没有另起一套后端逻辑，符合复用现有发布 API 的原则。

发现问题：

1. CLI 自研 tar 打包只支持短路径。
   - 位置：`cli/lib/core.js`
   - 说明：tar header 的文件名只写入 100 字节，深层目录或长文件名会失败。
   - 影响：AI 工具生成项目如果目录层级较深，CLI 发布会失败。
   - 是否阻塞部署：否，但属于下版高优先级修复项。

2. CLI 在额度不足时，错误信息可能混入项目修复建议。
   - 位置：`cli/lib/core.js`
   - 说明：403 额度错误仍可能拼接 `fixPrompt`。
   - 影响：用户可能误以为项目本身有问题，而不是账号额度问题。
   - 是否阻塞部署：否。

3. MCP npm 包结构未来可能漏依赖。
   - 位置：`mcp/package.json`
   - 说明：`files` 只包含 `bin`，但 MCP 代码引用了 CLI 的 `lib/core.js`。
   - 影响：当前 zip 包可用；如果未来发布 npm 包，可能出现缺文件。
   - 是否阻塞当前部署：否。

### 第三轮：前端首页、用户端和后台体验

审查范围：

- 首页营销表达。
- 登录注册。
- 用户端项目列表、详情抽屉、发布记录、表单记录、AI 发布页。
- 管理后台用户、项目、升级申请、反馈、表单记录。
- 文案是否面向非技术用户。

结果：

- 首页、用户端、后台整体已经比早期版本更统一。
- 用户端详情抽屉和后台详情抽屉方向正确，避免把详情堆在列表下方。
- AI 发布页已经具备发布口令和接入指令入口。

发现问题：

1. 用户端 AI 发布指令硬编码 API 地址。
   - 位置：`web/src/pages/UserDashboard.tsx`
   - 说明：指令里固定使用 `http://8.155.150.162`。
   - 影响：后续换域名、HTTPS、本地测试或多环境部署时，容易误导用户和 AI 工具。
   - 是否阻塞部署：否，但上线体验风险较高。

2. 首页标题仍存在移动端潜在换行风险。
   - 位置：`web/src/styles/home.css`
   - 说明：桌面截图正常，但固定分行和 `white-space: nowrap` 可能在移动端造成溢出或怪异换行。
   - 影响：移动端第一印象可能受影响。
   - 是否阻塞部署：否。

3. 用户端和后台的完整登录态视觉回归还需要更稳定的自动化方案。
   - 说明：未登录截图只能覆盖首页、登录页和跳转态；后台受 Basic Auth/API 权限限制，截图停留在加载或授权态。
   - 影响：后续前端大改时，仍需要补齐带登录态的自动化验收。
   - 是否阻塞部署：否。

### 第四轮：版本、打包与部署脚本

审查范围：

- v0.1.23 版本号。
- 站点包、服务端包、运维脚本包、CLI 包、MCP 包。
- 上传脚本、部署脚本、回滚脚本、清理脚本。

结果：

- 服务端版本为 `0.1.23`。
- CLI 版本为 `0.1.23`。
- 打包脚本已生成 5 个包：
  - `dist/demogo-site-preview.zip`
  - `dist/demogo-server-v0.1.23.zip`
  - `dist/demogo-ops-scripts-v0.1.23.zip`
  - `dist/demogo-cli-v0.1.23.zip`
  - `dist/demogo-mcp-v0.1.23.zip`

发现问题：

1. `npx demogo deploy` 的真实可用性依赖 npm 发布。
   - 说明：本地 zip 包可运行，但如果 npm 未发布，用户在真实 AI 工具里直接执行 `npx demogo deploy` 会失败。
   - 影响：对外宣传“一句话让 AI 发布”前，需要明确安装方式或先完成 npm 发布。
   - 是否阻塞部署：取决于是否马上对外宣传该能力。

## 问题汇总

### 阻塞部署

暂无。

### 建议下一版本优先修复

1. CLI 支持深层目录和长文件名。
2. AI 发布指令中的 API 地址改为动态生成或后端配置。
3. 明确“下线项目”与“表单额度释放”的产品规则，并在界面给出提示。
4. MCP npm 包结构补齐依赖，避免未来发布 npm 包时缺文件。
5. 建立带登录态的用户端和后台视觉回归测试。

### 观察项

1. 后端重复分析项目包会影响大包速度，后续可以优化性能。
2. 额度不足时 CLI 错误提示应更聚焦账号额度，不要混入项目修复建议。
3. 首页移动端标题需要继续做真实设备验收，避免奇怪换行。
4. `npx demogo deploy` 对外宣传前，需要确认 npm 发布或改成“安装后执行”的表达。

## 部署判断

v0.1.23 可以进入部署准备。当前问题不阻塞部署，但建议在下一个版本集中处理 AI 发布体验和 CLI/MCP 分发问题。
