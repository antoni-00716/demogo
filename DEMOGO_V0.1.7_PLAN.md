# DemoGo v0.1.7 版本计划

更新时间：2026-05-11

## 1. 版本定位

v0.1.7 不做大而全功能，重点是把 DemoGo 从“可用 MVP”升级为“准商业化地基版”。

本版本核心目标：

> MySQL 数据库地基 + 发布体验优化 + 项目检测增强 + 规则体检报告。

本版本不接大模型，不做表单托管，不做支付，不做完整后端应用发布。

## 2. 为什么这样安排

当前最关键的问题不是功能少，而是地基还不够商业化：

- 主数据还在 JSON 文件里；
- 后续套餐、表单、订单、统计都需要正式表结构；
- 用户发布失败时不够清楚；
- 活动报名页、预约页这类项目里的表单和本地 API 风险还不能明确提示；
- AI 能力应先从稳定的规则报告做起，避免一开始引入大模型成本和不确定性。

## 3. v0.1.7 范围

### P0：MySQL 数据库地基

目标：将核心业务数据从 JSON 主存储迁移到 MySQL。

必须完成：

- 建库建表脚本；
- 后端数据库连接配置；
- JSON 到 MySQL 迁移脚本；
- 迁移前备份；
- 迁移后校验；
- 回滚说明；
- 后端核心读写改为 MySQL。

第一批表：

- `users`
- `sessions`
- `plans`
- `demos`
- `demo_versions`
- `project_inspections`
- `audit_logs`

暂不落地：

- `forms`
- `form_fields`
- `form_submissions`
- `orders`
- `payments`
- `ai_reports`

这些表保留设计，后续版本再实现。

### P0：发布体验优化

目标：让非技术用户理解“发布会先检测，检测通过才发布”。

前端需要优化：

- 上传区说明当前支持项目类型；
- “开始发布”按钮文案说明自动检测；
- 检测失败时展示原因和建议；
- 发布成功后展示链接、有效期、项目类型、注意事项；
- 对“后端、数据库、Docker 暂不支持”做清楚提示。

### P0：项目检测增强

目标：让 DemoGo 能识别更多真实 AI 生成项目的问题。

后端检测结果应包含：

- 入口文件；
- 是否包含 `package.json`；
- 是否包含 build 命令；
- 可发布文件数；
- 可发布体积；
- 已忽略文件；
- 被阻止文件；
- 疑似表单字段；
- 疑似本地 API 调用。

重点检测：

- `<form>`；
- `input`、`textarea`、`select`；
- `fetch("/api/...")`；
- `axios.post("/api/...")`；
- 常见字段名：`name`、`phone`、`email`、`company`、`message`、`remark`。

### P1：规则体检报告

目标：不用大模型，也能生成用户能看懂的体检报告。

报告内容：

- 项目类型判断；
- 主要功能判断；
- 发布可行性；
- 发布后可能失效的功能；
- 表单/API 风险；
- 修复建议；
- 给 AI 编程工具的提示词。

示例：

```text
检测到这个项目可能是活动报名页，包含姓名、手机号、公司、备注等字段。
项目中调用了 /api/register，但当前 DemoGo 不托管自定义后端 API。
如果直接发布，页面可以打开，但报名数据可能无法保存。

你可以复制下面这段话给 AI 编程工具：
请把当前报名表的提交逻辑改为可配置接口，并预留接入 DemoGo 表单托管 API 的位置。
```

## 4. 不做事项

v0.1.7 不做：

- 大模型 AI 分析；
- 表单托管正式功能；
- 管理后台套餐调整；
- 在线支付；
- 订单系统；
- Docker 项目发布；
- 完整后端应用运行；
- 飞书/腾讯文档集成；
- 全量前端重构；
- 完整微服务拆分。

## 5. 技术实施顺序

建议顺序：

1. 确认 MySQL 版本、连接方式、数据库账号；
2. 编写 MySQL 建表脚本；
3. 编写数据库访问层；
4. 编写 JSON 到 MySQL 迁移脚本；
5. 本地用样例 JSON 测试迁移；
6. 改造后端核心读写；
7. 增强项目检测；
8. 增加规则体检报告；
9. 优化用户端展示；
10. 自测核心流程；
11. 打包部署；
12. 线上迁移和验证。

## 6. 验收标准

### 数据库

- 用户、Session、Demo、日志能从 MySQL 读取；
- 现有 JSON 数据能迁移；
- 迁移前后数量一致；
- 发布、更新、下线、恢复、删除操作都正常；
- 迁移失败有回滚方案。

### 发布流程

- 用户可注册登录；
- 可上传静态项目；
- 可上传常见前端源码项目；
- 可检测；
- 可发布；
- 可更新；
- 可下线/恢复/删除；
- Demo 链接可访问。

### 检测报告

- 能展示入口文件；
- 能展示 build 命令情况；
- 能展示表单字段；
- 能展示疑似本地 API；
- 能给出清楚的风险提示；
- 能生成给 AI 编程工具的修复提示。

## 7. 风险与控制

| 风险 | 控制方式 |
|---|---|
| 数据迁移出错 | 迁移前备份 JSON，迁移后校验数量和关键字段 |
| 数据库连接失败 | 保留部署前验证脚本 |
| 后端改动影响已有发布 | 先本地测试，再打包，再部署 |
| 检测规则误判 | 报告使用“可能”“疑似”，不强行阻止发布 |
| v0.1.7 范围膨胀 | 大模型、表单托管、套餐调整后置 |

## 8. 下一步

开发前需要确认：

- MySQL host、port、database、user；
- 是否允许创建新数据库；
- 线上 JSON 数据是否需要一次性迁移；
- 是否接受短暂停机迁移；
- v0.1.7 是否暂不处理套餐调整。
