# DemoGo v0.1.10 版本计划：试用转化闭环修复版

更新时间：2026-05-12

## 1. 版本定位

v0.1.10 不做大功能扩张，重点修复 v0.1.9 上线后暴露出的试用转化闭环问题。

一句话目标：

> 用户能申请升级，你能处理申请，Demo 链接能更自然传播，同时继续把老架构小步迁移到更清晰的新架构。

## 2. 产品问题

从产品经理、用户、运营三个角度看，当前核心问题是：

- 用户看到额度不足，但没有正式申请升级入口；
- 管理员能手动改套餐，但不知道哪些用户主动想升级；
- 复制转发文案使用浏览器弹窗，体验不专业；
- 后台缺少“待处理升级申请”这种运营待办；
- 新功能如果继续堆在旧代码里，会加重架构债。

## 3. P0 功能范围

### A. 用户申请升级套餐

- 用户端在“套餐额度”区域提供申请入口；
- 支持申请 `lite` 或 `pro`；
- 支持填写联系方式和申请说明；
- 同一用户同一时间只允许一个待处理申请；
- 提交后进入 `open` 状态；
- 写入审计日志。

### B. 管理后台处理升级申请

- 管理后台新增“升级申请”模块；
- 支持按用户、联系方式、说明搜索；
- 支持按申请套餐筛选；
- 支持按状态筛选；
- 管理员可开通或拒绝；
- 开通后自动更新用户套餐；
- 拒绝时支持记录原因；
- 处理动作写入审计日志。

### C. 转发文案体验修复

- 复制文案不再使用浏览器原生弹窗；
- 一键复制失败时显示页面内文案面板；
- 文案面板提供可选中文本框；
- 支持“选中文案”和“收起”；
- 保持文案不夸大 DemoGo 能力。

### D. 运营待办

- 管理后台显示待处理升级申请数；
- 用户端显示最近一次升级申请状态；
- 额度耗尽时提示申请升级。

## 4. 架构演进范围

v0.1.10 必须包含小步架构迁移，不做大重构。

本版本架构演进项：

- 新增正式表 `plan_upgrade_requests`；
- 新增 `server/src/services/plan-request-service.js`；
- MySQL 迁移和校验脚本支持升级申请表；
- smoke test 覆盖升级申请、重复申请拦截、后台审批、套餐生效；
- 页面复制逻辑从浏览器弹窗迁移为自有 UI 组件；
- 后续继续把 `server.js` 中的路由拆到 `routes/`。

参考方向：

- 学习 Vercel 的工程边界和测试意识；
- 学习 Coolify 的应用、数据库、部署任务边界；
- 学习 Dokploy 的 API 分组和任务抽象；
- 学习 CapRover 的简单后台和自动化运维思路。

## 5. 不纳入本版本

- 在线支付；
- 正式订单系统；
- 表单托管正式功能；
- 大模型 AI；
- Docker / 后端项目托管；
- 完整微服务拆分；
- 对象存储、CDN、OpenAPI、CLI。

## 6. 验收标准

- 用户能提交升级申请；
- 重复提交待处理申请会被阻止；
- 管理员能看到升级申请列表；
- 管理员能开通申请并自动更新用户套餐；
- 管理员能拒绝申请；
- 用户刷新后套餐额度生效；
- 转发文案不再出现浏览器弹窗；
- 本地 `npm run check` 通过；
- 本地 `npm run test:smoke` 通过；
- 部署前有备份、部署、验证、回滚步骤。
