# DemoGo v0.2.1 开发方案

## 目标

v0.2.1 是 v0.2.0 真实 AI 发布试用后的修正版，不改变产品边界，重点解决四类问题：

- AI 发布项目名过泛，例如在 DemoGo 目录发布后项目名变成 `demogo`。
- 自动表单识别过宽，把价格计算器、费用切换、模型参数等控件误判成报名/留言表单。
- 运营后台无法区分发布来源是网页上传、CLI、MCP 还是直接 Agent API。
- CLI 本机配置缺少自检和清除能力，排查体验不顺。

## 本版实现

- 项目名生成：优先使用用户明确填写的非泛化名称，其次使用页面 `<title>`、首个 `<h1>`，最后才使用压缩包或目录名。
- 发布来源：`/api/agent/deploy` 支持 `source` 字段、`X-DemoGo-Deploy-Source` 请求头和 User-Agent 识别。
- 表单判断：保留填写控件检测，但只有报名、预约、留资、留言类字段才自动开启 DemoGo 表单收集。
- CLI：升级到 `0.2.1`，增加 `demogo doctor` 和 `demogo config clear`。
- MCP：升级到 `0.2.1`，请求中标识来源为 `mcp`。
- 后台/用户端：项目详情展示发布方式。
- 安全补强：登录失败增加内存级限速，降低暴力尝试风险。
- AI 发布页：Token 按长期复用设计，复制发布指令不再绑定“重置口令”。
- 单 HTML 发布：`landing-page.html`、`home.html` 等单个 HTML 文件可直接作为首页发布。
- AI 指令：明确 CLI 不可用时要说明原因，再使用 MCP 或 Agent API 兜底。

## 不做内容

- 不做 Node/Python 后端托管。
- 不做数据库托管、支付、订单、登录系统后端。
- 不做 npm registry 正式发布流程。
- 不做三端前端大重构。
- 不改变 v0.2.0 已确认的 AI 工具发布主路线。

## 验收重点

- `/api/health` 返回 `0.2.1`。
- CLI/MCP/Skill 打包产物版本为 `0.2.1`。
- 泛化项目名会被页面标题或主标题替代。
- 价格计算器类控件不会自动开启表单收集。
- 报名/留言表单仍能自动开启收集。
- 后台能看到发布方式。
- 自动化检查、烟雾测试、前端构建通过。
